# Table: `user_auth_providers`

**Domain:** Auth
**Tenant-scoped:** No — global (user-level, not tenant-scoped)
**Prisma model:** `UserAuthProvider`
**Added:** 2026-04-09 · **Revised:** 2026-04-23 (see `authentication-strategy-v2.md`)

---

## Part 1 — Overview

Links a user to one or more authentication providers. This is the core table
that enables the multi-provider auth strategy: **Telegram, Facebook, and
Phone-OTP** as the three MVP providers. A user must link at least two of
them — enforced as a hard gate during onboarding.

A single user can have multiple rows — one per linked provider. A typical
merchant has two rows (Facebook + Phone, or Telegram + Phone, etc.), giving
them two independent recovery paths from day one.

### Why these three providers

`authentication-strategy-v2.md` picked them for Cambodia-specific reasons:

- **Telegram & Facebook:** the two daily-use identity platforms for
  Cambodian SMBs. Either one is a natural primary login.
- **Phone-OTP:** phone numbers are universal in Cambodia (tied to ID by
  law, used by ABA / Wing / every local banking flow). SMS OTP is a
  familiar, reliable recovery UX. Works unconditionally when both socials
  are lost.

Google and email were considered and rejected — Google has negligible
Cambodian SMB coverage, and email adoption is low enough that "email
recovery" becomes "forgot email password → stuck." If real demand appears
later, adding them back is a code + single-migration change.

### The "at least two linked" invariant

Every merchant, manager, platform admin, and platform staff account must
have **at least two `user_auth_providers` rows** at all times:

- Onboarding cannot mark `setup_progress.profile_completed_at` until two
  methods are linked → "Go Live" is gated on that.
- Unlinking a provider is rejected if it would leave the user with fewer
  than two linked methods. The UI shows "You need at least one backup login
  method before removing this one."

Frontline staff (KITCHEN_STAFF, SERVICE_STAFF) do NOT use this table —
they log in via PIN on a tenant-bound shared tablet. See the design
decision in Part 7.

See `design-discussions/authentication-strategy-v2.md` for the full decision
and use-case walkthroughs.

## Part 2 — CREATE TABLE

```sql
CREATE TYPE "AuthProvider" AS ENUM (
  'TELEGRAM',    -- primary login (OAuth-equivalent via Telegram Login Widget)
  'FACEBOOK',    -- primary login (OAuth 2.0)
  'PHONE'        -- recovery + standalone login via SMS OTP
);

CREATE TABLE user_auth_providers (
  id            TEXT PRIMARY KEY,
  user_id       TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  provider      "AuthProvider" NOT NULL,
  provider_id   TEXT NOT NULL,                         -- Telegram user ID, Facebook ASID, or phone (E.164)
  display_name  TEXT,                                  -- name from provider's profile (NULL for PHONE)
  avatar_url    TEXT,                                  -- profile pic URL (NULL for PHONE)
  metadata      JSONB,                                 -- raw provider response for audit (NULL for PHONE)
  linked_at     TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE (provider, provider_id)                       -- one XFOS account per provider identity
);

CREATE INDEX ON user_auth_providers (user_id);
```

## Part 3 — Column-by-column explanation

### `id` — TEXT PRIMARY KEY

System-internal identifier. cuid generated by the application. Never exposed
to users or external systems.

### `user_id` — TEXT NOT NULL, FK → users(id)

The XFOS user this provider is linked to. Multiple rows can share the same
`user_id` (one per linked provider). The FK cascades on delete — if the
user is hard-deleted, all their auth providers are removed.

**Why NOT NULL:** An auth provider link without a user is meaningless. Every
row must belong to exactly one user.

### `provider` — AuthProvider NOT NULL

Which authentication provider this row represents.

| Value | What it is | Flow |
|---|---|---|
| `TELEGRAM` | Telegram Login Widget | Login Widget embedded in the login page. User confirms via Telegram app; Telegram returns user data signed with the bot token. The app MUST verify the signature before trusting the payload. |
| `FACEBOOK` | Facebook Login | OAuth 2.0 via Facebook Login SDK. User taps "Continue with Facebook", Facebook confirms identity, returns access token + user ID. |
| `PHONE` | Phone-OTP via SMS | Not OAuth. User enters phone number; system sends a 6-digit OTP via SMS gateway (see `phone_otp_attempts`). User enters the code within 5 minutes. On success, `users.phone_verified = TRUE` and a PHONE row exists with `provider_id = users.phone`. |

**Why an enum instead of free-form TEXT:** Each provider requires a specific
integration (SDK, signature verification, or OTP workflow). Adding a new
provider requires code changes anyway — an enum keeps the allowed values
explicit and type-checked at both DB and TypeScript layers.

### `provider_id` — TEXT NOT NULL

The unique identifier for the user within the provider's system.

| Provider | What `provider_id` stores | Example |
|---|---|---|
| `TELEGRAM` | Telegram user ID (numeric string) | `"987654321"` |
| `FACEBOOK` | Facebook user ID (numeric string — **App-Scoped ID**, see design note) | `"10234567890123456"` |
| `PHONE` | The phone number in E.164 format (matches `users.phone`) | `"+85512345678"` |

**Why TEXT not BIGINT for numeric IDs:** Facebook and Google IDs are numeric
but can exceed JavaScript's `Number.MAX_SAFE_INTEGER`. Storing as TEXT avoids
precision loss and works uniformly across all providers.

**Why PHONE `provider_id` mirrors `users.phone`:** The canonical phone
column lives on `users` (one per user, UNIQUE, E.164, with verification
flags). The PHONE provider row's `provider_id` is a denormalized copy used
for fast lookup via the `UNIQUE (provider, provider_id)` index during OTP
login. The application keeps them in sync: any update to `users.phone`
also updates the corresponding PHONE row (and resets `phone_verified`
until a new OTP is completed).

**Facebook `provider_id` is App-Scoped (ASID):** the ID Facebook returns is
scoped to the specific Facebook App. **Do not change the Facebook App
post-launch** without a migration plan — all stored FACEBOOK provider_ids
would become invalid, and the `business_mapping` API would be needed to
translate old IDs to new ones.

### `display_name` — TEXT, nullable

The user's display name as returned by the provider's profile endpoint.

- Facebook: the user's Facebook name
- Telegram: `first_name` + `last_name`
- Google: the Google profile name
- Email: typically NULL (no profile to fetch)

**Why nullable:** Not all providers return a name (email doesn't), and the
user might have privacy settings that hide their name. The application uses
this to pre-populate `users.full_name` on first link if `full_name` is NULL.

**Why stored here instead of only on `users`:** A user might have different
names across providers (Facebook name vs Telegram name vs Google name). This
preserves the original name from each provider for audit, while `users.full_name`
holds the canonical name the user chose for XFOS.

### `avatar_url` — TEXT, nullable

Profile picture URL from the provider. Used to populate `users.avatar_url`
on first link. Same rationale as `display_name` — different providers may
return different profile pictures.

**Note:** These URLs point to the provider's CDN (e.g. `graph.facebook.com`,
`t.me`, `lh3.googleusercontent.com`). They may expire or change if the user
updates their profile picture. The application should cache/proxy avatars
if long-term persistence is needed.

### `metadata` — JSONB, nullable

Raw response payload from the provider during the OAuth flow. Stored for
audit and debugging purposes.

**What goes in here:**
- Facebook: `{ "id": "...", "name": "...", "email": "...", "picture": { ... } }`
- Telegram: `{ "id": ..., "first_name": "...", "username": "...", "auth_date": ... }`
- Google: the decoded ID token claims
- Email: typically NULL or `{}`

**Why JSONB not JSON:** JSONB is indexable and comparaable in PostgreSQL.
Even though we don't index this column, JSONB is the standard choice for
structured data in Postgres — it normalizes on write (deduplicates keys,
removes whitespace) and is more efficient to query.

**Privacy note:** This may contain PII from the provider (email, name, photo).
Ensure it's covered by the data retention policy and scrubbed when the user
is deleted (`users.status = DELETED`).

### `linked_at` — TIMESTAMP(3) NOT NULL, default now()

When this provider was linked to the user's account. Not the same as the
user's `created_at` — a provider can be linked at any time (onboarding or
later from account settings).

**Why no `updated_at`:** This row is essentially immutable. You link a
provider or you unlink it (delete the row). There's no "update" to an auth
provider link — if the user's Facebook changes, we fetch fresh data from
Facebook on each login, not from this table.

## Part 4 — Indexes

### `UNIQUE (provider, provider_id)`

**What it enforces:** One XFOS account per provider identity. A Facebook
account can only be linked to one XFOS user.

**What query it serves:**
```sql
-- Login: find the XFOS user for this Facebook account
SELECT user_id FROM user_auth_providers
WHERE provider = 'FACEBOOK' AND provider_id = '10234567890123456';
```

This is the **primary lookup during login** — it must be fast. The unique
constraint creates an implicit B-tree index on `(provider, provider_id)`.

**What it prevents:**
- Two XFOS accounts linked to the same Facebook → rejected.
- Account hijacking: if attacker creates a new XFOS account with a
  Facebook ID that's already linked → blocked by the constraint.

### `INDEX (user_id)`

**What query it serves:**
```sql
-- Account settings: show all linked providers for this user
SELECT provider, display_name, linked_at FROM user_auth_providers
WHERE user_id = 'clx8h2k9m...';

-- Check if user has a backup provider (enforce 2+ linked)
SELECT COUNT(*) FROM user_auth_providers
WHERE user_id = 'clx8h2k9m...';
```

## Part 5 — Relationships

### `user_id` → `users(id)` ON DELETE CASCADE

If a user is hard-deleted from the `users` table, all their auth provider
links are removed. This ensures no orphaned auth records.

**Why CASCADE not SET NULL:** An auth provider link without a user is
meaningless — it should never exist. If the user is gone, the links go with
them.

**Note on soft-delete:** In practice, users are soft-deleted via
`users.status = DELETED`, not hard-deleted. The CASCADE is a safety net
for the rare case of actual row deletion (e.g., GDPR right-to-be-forgotten).

### No FK to `tenants`

This table is **not tenant-scoped**. Auth providers are user-level — a
user's Facebook account is the same regardless of which tenant they belong
to. `user_roles` handles the tenant association, not this table.

## Part 6 — Real-world usage scenarios

### Scenario 1 — Tenant owner signs up with Facebook, links phone as backup

A noodle stall owner in Phsar Thmei taps "Continue with Facebook" on the
XFOS registration page.

```
1. Facebook OAuth flow → returns user ID, name, email (if scope granted), profile pic
2. Check: SELECT FROM user_auth_providers WHERE provider='FACEBOOK' AND provider_id='10234...'
3. Not found → new user
4. Create users row:
     { email: "sokha@gmail.com",       -- captured from FB, email_verified = TRUE
       full_name: "សុខា",
       avatar_url: "https://graph.facebook.com/...",
       status: 'PENDING' → 'ACTIVE' on provider confirmation }
5. Create user_auth_providers row #1:
     { user_id: user_001, provider: FACEBOOK, provider_id: "10234...",
       display_name: "សុខា", metadata: { ...raw FB response } }

-- Onboarding enforces "at least 2 methods linked" before Go Live:
6. Wizard shows: "One more step — protect your account. Add a backup login."
7. Owner taps "Add phone number":
     - Enters 012 345 678 → app normalizes to +85512345678
     - App writes users.phone = '+85512345678'
     - App calls SMS gateway; sends 6-digit OTP
     - phone_otp_attempts row created with otp_hash, expires_at = NOW() + 5 min
8. Owner enters OTP code. App verifies:
     - users.phone_verified = TRUE, phone_verified_at = NOW()
     - Create user_auth_providers row #2:
         { user_id: user_001, provider: PHONE, provider_id: "+85512345678" }
9. Two methods linked. setup_progress.profile_completed_at becomes settable.
```

**Result:** 1 user, 2 auth provider rows (FACEBOOK + PHONE). Can log in
with either — Facebook OAuth for daily convenience, phone-OTP for recovery.

### Scenario 2 — Facebook locked, owner recovers via phone-OTP

The same owner's Facebook account gets locked (Khmer name policy
violation — common in Cambodia). They can't use Facebook login.

```
1. Owner goes to XFOS login page → taps "Use phone number"
2. Enters +85512345678
3. App looks up: SELECT FROM users WHERE phone = '+85512345678' AND phone_verified = TRUE
   → finds user_001
4. App sends 6-digit OTP via SMS gateway
   (phone_otp_attempts row created, rate-limited to 1 per 60s per phone)
5. Owner enters OTP within 5 minutes → verified
6. App verifies the PHONE user_auth_providers row exists, status = ACTIVE, issues JWT
```

**Result:** Back in the system within 2 minutes. No support call, no data
loss, business continues. This is why the two-of-three linked-methods
rule is a hard gate — recovery must work without human intervention.

### Scenario 2b — Both socials lost, phone-OTP is the unconditional recovery

Bopha's Facebook is locked AND her Telegram account is on a stolen phone.
Her SIM card is replaced with a new one, same number.

```
1. Taps "Use phone number" → enters same phone
2. SMS arrives on the new phone (same number)
3. Enters OTP → logged in
4. From account settings, unlinks the dead Facebook + any Telegram row.
   Re-links fresh providers at her leisure (must keep at least 2 linked).
```

Phone is the unconditional recovery lever because phone-number portability
is the most reliable identity anchor in Cambodia (number follows the
person across SIM changes).

### Scenario 3 — Platform admin uses Telegram + Phone

An XFOS platform admin is added by another admin via the Platform Admin
portal.

```
1. Admin invitation sent (via Telegram or Messenger — out-of-band for platform users)
2. New admin taps "Continue with Telegram" → signs in
3. Required to add a second method. Picks Phone.
4. Resulting auth_providers rows:
     { provider: TELEGRAM, provider_id: "15234..." }
     { provider: PHONE,    provider_id: "+85598765432" }
```

**Result:** 2 providers, both independent. Same model as merchants — no
special email+password path for platform admins.

### Scenario 4 — Preventing duplicate accounts

A user tries to link a Facebook account that's already linked to another user.

```
1. User B tries to link Facebook ID "10234..." (already linked to User A)
2. INSERT INTO user_auth_providers ... → UNIQUE violation on (FACEBOOK, "10234...")
3. System rejects: "This Facebook account is already linked to another XFOS account."
```

The unique constraint prevents one provider identity from being spread
across multiple XFOS accounts.

## Part 7 — Design decisions

### Why a separate table instead of columns on `users`

Alternative: add `facebook_id`, `telegram_id`, `google_id` columns
directly on `users`.

**Rejected because:**
- Adding a new provider requires a schema migration (add column). With a
  separate table, adding a provider is just a new enum value.
- Can't easily query "how many providers does this user have?" without
  checking N nullable columns.
- No clean place to store per-provider metadata (display name, avatar, raw
  response) — you'd need `facebook_name`, `telegram_name`, etc.
- The junction table pattern is the industry standard for multi-provider auth
  (used by Auth0, Firebase, Supabase Auth).

### Why `provider_id` is TEXT not a provider-specific type

All provider IDs (Facebook numeric ID, Telegram numeric ID, Google sub,
email address) are stored as TEXT in the same column. This keeps the schema
uniform — the application layer knows how to interpret `provider_id` based
on the `provider` enum.

Alternative: separate `facebook_id BIGINT`, `telegram_id BIGINT`, etc. —
rejected for the same reasons as adding columns to `users`.

### Why no `unlinked_at` (soft delete for unlinking)

When a user unlinks a provider, the row is hard-deleted. There's no
`unlinked_at` column or soft-delete pattern.

**Rationale:** Unlinking a provider is rare and low-risk. If the user
re-links the same provider later, a new row is created. The audit trail
for "who linked/unlinked what" is in `audit_logs`, not in this table.

### Why no `tenant_id`

Auth providers are user-level, not tenant-level. A user's Facebook account
is the same regardless of which XFOS tenant they work for. Tenant
association is handled by `user_roles`, not by auth providers.

### Why PIN is not an `AuthProvider` value

Frontline staff (KITCHEN_STAFF, SERVICE_STAFF) log in via a 4-6 digit PIN
on a tenant-bound shared tablet (see `authentication-strategy-v2.md`
Scenarios 8 & 9 + the future `user_pins` table). This is deliberately
scoped differently:

- **Tenant-specific, not user-global.** A PIN is meaningful only at a
  specific restaurant's kitchen or counter tablet. The same person
  working at two tenants has two independent PINs, not one shared one.
- **Tablet-bound.** The tablet is pre-configured with `tenant_id` and a
  `device_role` (KITCHEN or SERVICE). The PIN identifies the person
  *within that tenant's device context*. Moving the tablet to another
  tenant requires re-provisioning.
- **No external provider.** PIN is self-contained — no OAuth, no SMS
  gateway, no third-party trust. Just a hashed digit string on a tenant
  table.

These three properties map cleanly to a tenant-scoped `user_pins` table,
not to this user-global `user_auth_providers` table. Putting PIN here
would force awkward NULL `tenant_id` semantics on a row that is only
meaningful with a tenant. The split keeps both tables honest.

### Why Phone-OTP uses `user_auth_providers`, not a dedicated table

Phone-OTP shares three properties with the OAuth providers that make it a
natural fit here:

- **User-global, not tenant-scoped.** A phone number identifies a person
  across every tenant they work at.
- **Fits the same lookup shape.** Login via phone uses the same
  `WHERE provider = ? AND provider_id = ?` query pattern as Telegram or
  Facebook login — just with a different provider value.
- **Subject to the "2+ linked" invariant.** Counting linked providers for
  the onboarding gate is a single `SELECT COUNT(*) WHERE user_id = ?`.
  Putting phone in a separate table would complicate the invariant.

The SMS OTP workflow (send code, verify, rate-limit) lives in a separate
`phone_otp_attempts` table — but the *identity linking* (this phone belongs
to this user) lives here, alongside the other providers.

### Why no `access_token` / `refresh_token` columns

The original design considered storing OAuth access/refresh tokens for
ongoing API access (e.g., posting to the user's Facebook page). This was
removed because:

- XFOS doesn't need ongoing API access to user's social accounts.
- Storing OAuth tokens is a security liability (if the DB is breached,
  attacker gets access to users' Facebook/Google accounts).
- The OAuth flow is used only for **authentication** (prove identity),
  not for **authorization** (act on behalf of the user).
- If ongoing access is ever needed (e.g., posting to Facebook pages for
  marketing), it should be a separate `social_connections` table with
  encrypted tokens and explicit scopes, not mixed into the auth table.

## Part 8 — Related tables

| Table | Relationship | How |
|---|---|---|
| `users` | Parent | `user_id` FK → `users(id)`. CASCADE on delete. |
| `user_roles` | Sibling | Both reference `users`. Auth providers determine HOW the user logs in; roles determine WHAT they can do. |
| `refresh_tokens` | Sibling | After successful auth (via any provider), a refresh token is issued. The refresh token doesn't know which provider was used — it's provider-agnostic. |
| `invitations` | Flow dependency | When an invitation is accepted, the accepting user links a provider. The invitation's `channel` (telegram, facebook_messenger, email-reserved) hints at which provider they'll use, but doesn't enforce it. |
| `phone_otp_attempts` | Workflow dependency (PHONE only) | When a user initiates phone-OTP login or adds phone as a provider, a `phone_otp_attempts` row tracks the OTP send, rate limit, and verification. On success, a PHONE row in this table is created or confirmed. |
| `users.phone` | Mirror | The PHONE provider row's `provider_id` mirrors `users.phone`. Canonical phone lives on `users`; this table holds the auth-linking side. |
| `user_pins` (future, tenant-scoped) | Sibling (different scope) | Kitchen/service staff PINs live there, not here — see "Why PIN is not an AuthProvider value" in Part 7. |
| `audit_logs` | Audit trail | Provider linking/unlinking events are recorded in `audit_logs` with `action = 'user.provider_linked'` or `'user.provider_unlinked'`. |
