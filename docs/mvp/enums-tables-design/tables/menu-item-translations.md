# Table Reference: `menu_item_translations`

| Property | Value |
|---|---|
| **Domain** | Catalog |
| **Tenant-scoped** | Yes (denormalized -- C1 finding) |
| **Prisma model** | `MenuItemTranslation` |
| **Mapped table** | `menu_item_translations` |
| **Soft delete** | No (cascades with parent item) |

---

## Part 1: Overview

`menu_item_translations` stores the localized name and description for each
menu item. Each row holds one locale's translation of a single item. The
constraint `UNIQUE(item_id, locale)` ensures at most one translation per locale
per item.

**Why this table exists:**

This is the table that makes XFOS menus bilingual. When a customer in Phnom
Penh scans a QR code at a noodle stall, the storefront queries this table to
display "គុយទាវសាច់គោ" instead of "Beef Noodle Soup" -- or both, depending on
the customer's locale preference.

Unlike `menu_category_translations` (which only has `name`), this table also
has a `description` column. Items benefit from descriptions ("Slow-grilled pork
ribs with tamarind glaze, served with sticky rice") that help customers make
ordering decisions.

**Denormalized `tenant_id` (C1):** Like `menu_category_translations`, this
table carries its own `tenant_id` even though the parent `menu_items` row
already has one. This is the C1 finding from the schema stress test: every
tenant-scoped child table must carry its own `tenant_id` for query performance
and defense-in-depth via parity triggers.

---

## Part 2: CREATE TABLE

```sql
CREATE TABLE menu_item_translations (
  id          TEXT PRIMARY KEY,
  tenant_id   TEXT NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  item_id     TEXT NOT NULL REFERENCES menu_items(id) ON DELETE CASCADE,
  locale      "Locale" NOT NULL,
  name        TEXT NOT NULL,
  description TEXT,
  UNIQUE (item_id, locale)
);

CREATE INDEX ON menu_item_translations (tenant_id);
```

**Parity trigger** (from `20260410_mvp_hardening.sql`):

```sql
CREATE OR REPLACE FUNCTION menu_item_translations_tenant_parity() RETURNS trigger AS $$
DECLARE parent_tenant TEXT;
BEGIN
  SELECT tenant_id INTO parent_tenant FROM menu_items WHERE id = NEW.item_id;
  IF parent_tenant IS NULL THEN
    RAISE EXCEPTION 'menu_item_translations: parent item % not found', NEW.item_id;
  END IF;
  IF parent_tenant <> NEW.tenant_id THEN
    RAISE EXCEPTION 'menu_item_translations: tenant_id mismatch (row=%, parent=%)',
      NEW.tenant_id, parent_tenant;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER menu_item_translations_tenant_parity_trg
  BEFORE INSERT OR UPDATE OF tenant_id, item_id ON menu_item_translations
  FOR EACH ROW EXECUTE FUNCTION menu_item_translations_tenant_parity();
```

## Part 3: Column-by-Column

### `id` -- TEXT PRIMARY KEY

| Property | Value |
|---|---|
| Type | `TEXT` |
| Nullable | No |
| Default | Application-generated `cuid()` |
| Constraints | `PRIMARY KEY` |

Unique identifier for this translation row. Generated by the application via
Prisma's `@default(cuid())`.

**Why a surrogate key:** Same rationale as `menu_category_translations` --
XFOS standardizes on cuid PKs for every table. The natural key
`(item_id, locale)` is enforced separately as a UNIQUE constraint.

---

### `tenant_id` -- TEXT NOT NULL

| Property | Value |
|---|---|
| Type | `TEXT` |
| Nullable | No |
| Default | None -- set by application |
| Constraints | `NOT NULL`, `REFERENCES tenants(id) ON DELETE CASCADE` |

Denormalized tenant identifier. MUST match the `tenant_id` of the parent
`menu_items` row. Enforced by the `menu_item_translations_tenant_parity`
trigger.

**Why denormalize (C1):** This table is the most-read translation table in the
entire platform. Every storefront menu page load queries it. The denormalized
`tenant_id` avoids a JOIN to `menu_items` just to enforce tenant scoping:

```sql
-- Direct filter: no JOIN needed for tenant isolation
SELECT mit.name, mit.description
FROM menu_item_translations mit
WHERE mit.tenant_id = 'clxyz123abc'
  AND mit.item_id = 'clitem-001'
  AND mit.locale = 'kh';
```

Without the denormalized column, this query would need to JOIN to `menu_items`
to verify tenant ownership -- adding latency to the most critical user-facing
query in the system.

---

### `item_id` -- TEXT NOT NULL

| Property | Value |
|---|---|
| Type | `TEXT` |
| Nullable | No |
| Default | None -- set by application |
| Constraints | `NOT NULL`, `REFERENCES menu_items(id) ON DELETE CASCADE`, part of `UNIQUE(item_id, locale)` |

The parent menu item this translation belongs to.

**Why ON DELETE CASCADE:** When an item is hard-deleted (via tenant cascade),
its translations are cleaned up automatically. For normal operations, items are
soft-deleted (`deleted_at`), not hard-deleted, so this cascade rarely fires.
When an item is soft-deleted, its translations remain in the database -- they
become accessible again if the item is restored.

---

### `locale` -- "Locale" NOT NULL

| Property | Value |
|---|---|
| Type | `"Locale"` (Postgres enum: `'en'`, `'kh'`) |
| Nullable | No |
| Default | None -- set by application |
| Constraints | `NOT NULL`, part of `UNIQUE(item_id, locale)` |

Locale code. MVP supports `'en'` (English) and `'kh'` (Khmer).
Stored as a Postgres `"Locale"` enum and also validated by a Zod enum at the
API boundary.

**The `UNIQUE(item_id, locale)` constraint means:**

- Each item has at most one English translation and at most one Khmer
  translation.
- Attempting to insert a second Khmer translation for the same item fails with
  a unique constraint violation.
- The merchant portal uses `INSERT ... ON CONFLICT (item_id, locale) DO UPDATE`
  for upsert semantics.

---

### `name` -- TEXT NOT NULL

| Property | Value |
|---|---|
| Type | `TEXT` |
| Nullable | No |
| Default | None -- set by application |
| Constraints | `NOT NULL` |

The localized display name of the item. This is the primary text customers see
on the storefront. Examples from Cambodian food businesses:

| Locale | Name |
|---|---|
| `en` | Beef Noodle Soup |
| `kh` | គុយទាវសាច់គោ |
| `en` | Brown Sugar Milk Tea |
| `kh` | តែទឹកដោះគោស្ករត្នោត |
| `en` | Grilled Pork Skewers (3 pcs) |
| `kh` | សាច់ជ្រូកអាំងចង្កាក់ (៣ដុំ) |
| `en` | Fried Rice with Egg |
| `kh` | បាយឆាស៊ុត |

**Why NOT NULL:** A translation row without a name is pointless. If the name is
not available in a locale, the row should not exist -- the application falls
back to the `en` translation.

**Snapshot behavior:** When an order is placed, this name is copied into
`order_items.item_name`. The snapshot means renaming the item later does not
alter historical orders. The customer's receipt always shows the name as it was
at order time.

---

### `description` -- TEXT (nullable)

| Property | Value |
|---|---|
| Type | `TEXT` |
| Nullable | Yes |
| Default | `NULL` |
| Constraints | None |

Optional detailed description of the item. Displayed below the item name on the
storefront. Useful for items that benefit from explanation:

| Locale | Description |
|---|---|
| `en` | Slow-grilled pork ribs marinated in tamarind and lemongrass, served with sticky rice and dipping sauce |
| `kh` | ឆ្អឹងជំនីជ្រូកអាំងលាយទឹកអម្ពិលនិងស្លឹកគ្រៃ រួមជាមួយបាយដំណើបនិងទឹកជ្រលក់ |
| `en` | Classic Cambodian breakfast noodle soup with your choice of beef or pork |
| `kh` | គុយទាវពេលព្រឹកតាមបែបខ្មែរ មានសាច់គោ ឬសាច់ជ្រូក |

**Why nullable:** Many items are self-explanatory ("Iced Coffee", "Spring
Rolls") and do not need descriptions. Forcing a description would burden
merchants with unnecessary data entry. The storefront omits the description
block when the value is NULL.

**Why this column exists on `menu_item_translations` but not on
`menu_category_translations`:** Items are products -- customers need details
to decide what to order. Categories are navigation -- "Soups", "Drinks" -- and
do not require elaboration.

## Part 4: Indexes

### `menu_item_translations_tenant_idx` -- `(tenant_id)`

```sql
CREATE INDEX ON menu_item_translations (tenant_id);
```

**What queries it serves:** The primary query -- loading all item translations
for a tenant in a specific locale. This is the storefront's hot path.

**Example:**

```sql
-- Storefront: load all item names + descriptions for a tenant in Khmer
SELECT mit.item_id, mit.name, mit.description
FROM menu_item_translations mit
WHERE mit.tenant_id = 'clxyz123abc'
  AND mit.locale = 'kh';
```

This index makes the `tenant_id` filter an index scan. Given the small number
of items per tenant (typically 10-80), the `locale` filter on the remaining
rows is trivial.

### `menu_item_translations_item_id_locale_key` -- `UNIQUE(item_id, locale)`

```sql
UNIQUE (item_id, locale)
```

**What queries it serves:** Both a constraint and an implicit unique index.

1. **Single-item translation lookup.** The merchant portal edit screen loads one
   item's translations:

   ```sql
   SELECT mit.locale, mit.name, mit.description
   FROM menu_item_translations mit
   WHERE mit.item_id = 'clitem-beef-soup';
   ```

   The unique index on `(item_id, locale)` makes this an efficient range scan
   on `item_id`.

2. **Upsert on save.** The merchant portal save action uses conflict resolution:

   ```sql
   INSERT INTO menu_item_translations (id, tenant_id, item_id, locale, name, description)
   VALUES ('clnew123', 'cltenant456', 'clitem789', 'kh', 'គុយទាវសាច់គោ', 'គុយទាវពេលព្រឹក')
   ON CONFLICT (item_id, locale) DO UPDATE
     SET name = EXCLUDED.name, description = EXCLUDED.description;
   ```

3. **Storefront item detail page.** Loading a single item's Khmer translation
   is a point lookup on this index.

## Part 5: Relationships

### Parent: `tenants`

```
menu_item_translations.tenant_id → tenants.id (ON DELETE CASCADE)
```

Denormalized FK for tenant isolation. Tenant deletion cascades through.

### Parent: `menu_items`

```
menu_item_translations.item_id → menu_items.id (ON DELETE CASCADE)
```

The item this translation belongs to. Hard-deleting the item removes all its
translations. (Normal item removal is soft delete via `deleted_at`, which does
not trigger the cascade.)

### Parity trigger: `menu_item_translations_tenant_parity`

Fires `BEFORE INSERT OR UPDATE OF tenant_id, item_id`. Looks up the parent
item's `tenant_id` and raises an exception if they do not match. This is the
database-level safeguard against cross-tenant data bleed.

### Indirect: `order_items`

When an order is placed, `menu_item_translations.name` (in the customer's active
locale) is copied into `order_items.item_name`. After this snapshot, the
translation and the order item are independent -- changing or deleting the
translation does not affect the order.

## Part 6: Real-World Usage Scenarios

### Scenario 1: Noodle stall creates items with bilingual translations

"Kuy Teav Phnom Penh" creates their menu. For each item, the merchant enters
both English and Khmer. Here is one item:

**`menu_items` row:**

| Column | Value |
|---|---|
| id | `clitem-ktpp-beef` |
| tenant_id | `cltenant-ktpp` |
| category_id | `clcat-soups` |
| base_price_cents | 300 |
| currency | `USD` |

**`menu_item_translations` rows:**

| locale | name | description |
|---|---|---|
| `en` | Beef Noodle Soup | Classic kuy teav with slow-simmered beef broth, rice noodles, and fresh herbs |
| `kh` | គុយទាវសាច់គោ | គុយទាវសាច់គោទឹកស៊ុបប្រហិត ជាមួយនំបញ្ចុកនិងបន្លែស្រស់ |

When a Khmer-speaking customer scans the QR code, the storefront shows
"គុយទាវសាច់គោ" with the Khmer description. When an English-speaking tourist
visits, they see "Beef Noodle Soup" with the English description.

### Scenario 2: Bubble tea chain with English-primary items

"BoBa Express" in the central business district primarily serves English-speaking
customers. They create items with English only and add Khmer later:

**Week 1 -- English only:**

| item_id | locale | name | description |
|---|---|---|---|
| `clitem-brown-sugar` | `en` | Brown Sugar Milk Tea | Taiwan-style brown sugar boba with fresh milk |

**Week 3 -- Khmer added:**

| item_id | locale | name | description |
|---|---|---|---|
| `clitem-brown-sugar` | `en` | Brown Sugar Milk Tea | Taiwan-style brown sugar boba with fresh milk |
| `clitem-brown-sugar` | `kh` | តែទឹកដោះគោស្ករត្នោត | បូបាស្ករត្នោតតាមបែបតៃវ៉ាន់ជាមួយទឹកដោះគោស្រស់ |

The merchant portal's onboarding wizard tracks this gap. The `setup_progress`
table's `translations_complete` flag transitions from `FALSE` to `TRUE` when
every visible item has both `en` and `kh` translations.

### Scenario 3: BBQ restaurant updates descriptions seasonally

"Angkor BBQ" changes the description of their signature dish for the holiday
season:

**Normal description:**

| locale | description |
|---|---|
| `en` | Slow-grilled pork ribs with tamarind glaze |
| `kh` | ឆ្អឹងជំនីជ្រូកអាំងទឹកអម្ពិល |

**Holiday update (upsert):**

```sql
INSERT INTO menu_item_translations (id, tenant_id, item_id, locale, name, description)
VALUES ('clnew-holiday', 'cltenant-angkor', 'clitem-ribs', 'en',
        'Signature Pork Ribs',
        'Holiday special: slow-grilled pork ribs with tamarind glaze and cranberry compote')
ON CONFLICT (item_id, locale) DO UPDATE
  SET name = EXCLUDED.name, description = EXCLUDED.description;
```

The name and description are updated atomically. Existing orders keep their
snapshotted names. The storefront reflects the new description on the next load.

## Part 7: Design Decisions

### Denormalized `tenant_id` (C1 stress test finding)

Same rationale as `menu_category_translations`, but amplified by the fact that
this table is read more frequently. Every storefront page load queries item
translations. The denormalized `tenant_id`:

1. **Eliminates a JOIN** on the platform's hottest read path.
2. **Provides a parity trigger** that catches cross-tenant bugs before they
   reach disk. The trigger checks that `NEW.tenant_id` matches the parent
   item's `tenant_id` on every INSERT and on every UPDATE that touches
   `tenant_id` or `item_id`.

### `UNIQUE(item_id, locale)` -- one translation per locale per item

This is the i18n model's core invariant. Without it, the application would need
to deduplicate translations at read time, which is error-prone and wasteful.
The constraint also powers the upsert pattern: `INSERT ... ON CONFLICT
(item_id, locale) DO UPDATE` -- a single atomic operation for creating or
updating a translation.

### `description` is nullable; `name` is not

Every translation must have a name (the item must be identifiable). Descriptions
are optional because:

- Many items are self-explanatory by name alone ("Iced Coffee", "French Fries").
- Requiring descriptions would slow down menu creation for stalls with 20+
  items.
- The storefront UI gracefully omits the description block when NULL.

### No `created_at` / `updated_at` timestamps

Same reasoning as `menu_category_translations`: translation rows are
lightweight satellite data. The parent item's `updated_at` is sufficient for
cache invalidation. Adding timestamps would increase write amplification for
every translation upsert, with minimal benefit.

### No soft delete

Translations are trivially re-creatable (they are just text strings keyed by
locale). Adding soft delete would mean every read query needs
`WHERE deleted_at IS NULL`, which adds complexity to the most-read table in
the catalog domain. When a translation is deleted, it is hard-deleted. The
parent item's soft delete status is independent -- if an item is soft-deleted,
its translations persist and become visible again on restore.

### Snapshot into `order_items.item_name` at order time

When a customer places an order, the current `name` value (in their active
locale) is copied into `order_items.item_name`. This snapshot ensures:

- A receipt from Tuesday still reads "Beef Noodle Soup" even if the merchant
  renames it to "Kuy Teav Special" on Wednesday.
- The kitchen ticket shows the name as it was when the customer ordered, not
  the merchant's latest edit.
- Deleting or updating a translation never rewrites order history.

This is enforced by the application layer at order creation time, not by a
database trigger.
