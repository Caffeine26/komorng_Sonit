## Part 1 — Overview

The `orders` table is the central record of the entire XFOS platform. Every dollar of revenue, every kitchen ticket, every payment, and every customer interaction ultimately traces back to a row in this table. It is the single source of truth for "a customer wants food."

An order represents a discrete request from a customer: "I want these items, at this price, for this location." Once created, an order is **immutable in its content** -- the items, prices, and totals do not change. What changes is the order's **status**, which flows through a pure lifecycle state machine driven by kitchen events: `SUBMITTED -> PREPARING -> READY -> COMPLETED` (or `CANCELLED`).

**The order record only exists after the payment gate.** There is no "waiting for payment" state on orders -- payment is tracked entirely by `BillStatus` + `PaymentStatus`. The moment an order row exists in the database, it is already accepted. See `design-discussions/order-status-redesign.md` for the full rationale.

The most important behavioral split in the system is encoded on this table:

- **`pay_timing = PAY_BEFORE`**: Payment happens first. Only after payment succeeds is the order record created (status `SUBMITTED`). A kitchen ticket is created at the same time. This is how bubble tea shops, coffee kiosks, and food courts work.
- **`pay_timing = PAY_AFTER`**: The order record is created immediately when the customer taps "Place Order" (status `SUBMITTED`). A kitchen ticket is created at the same time. Payment happens later when the customer requests the bill. This is how noodle stalls, bars, and traditional restaurants work.

In both cases, the initial order status is always `SUBMITTED`. The `pay_timing` column determines **when** the order is created relative to payment, not what status it starts in.

This column (`pay_timing`) is snapshotted from `tenant_settings.pay_timing` at order creation time, but can be overridden per-order by the merchant (e.g., granting pay-after treatment to a regular customer at a normally pay-before stall).

### Order number generation

The `order_number` column (e.g., `LB-042`) is generated by the `allocate_order_number(tenant_id)` Postgres function, which atomically increments a **daily-reset** counter in the `tenant_sequences` table and formats the result as `{tenants.code_prefix}-{zero_padded_counter}`. The counter resets to `1` at the tenant's local midnight (from `tenant_settings.timezone`). Runs in the same transaction as order creation — either both succeed or both roll back. No Redis, no split-brain risk.

Because the counter resets daily, `order_number` is **not unique across days for the same tenant**. Today's `LB-042` and yesterday's `LB-042` are different orders. Uniqueness is scoped to a tenant-day, not enforced at the DB level — the atomic allocator guarantees no same-day collisions, and cross-day "collisions" are intentional (that is what daily reset means). Historical lookups by `order_number` require a date filter; internal joins use `orders.id` (cuid, globally unique).

The format, daily-reset rationale, and industry precedent are documented in `design-discussions/order-numbering-strategy.md`.

### The `order_token` column — the security gate

The `order_token` is an unguessable, globally unique string used as the public identifier for the order status page. When a customer finishes ordering, they receive a URL like `xfos.com/s/{order_token}`. This token is safe to share (it does not reveal the tenant ID, the internal order ID, or the order number). It is cryptographically random and cannot be enumerated.

Critically: **`order_token` is the ONLY access credential for an order**. Guessing or knowing an `order_number` (like `LB-042`) gets an attacker nothing — it is a display label, not an authorization token. This separation is what makes it safe to use a short, human-friendly `order_number` everywhere.

## Part 2 — CREATE TABLE

```sql
-- Why: a single CANCELLED status conflates very different operational situations.
-- Captured separately so dashboards and merchant alerts can segment cleanly.
-- See enums/order-cancellation-reason.md.

  'CUSTOMER_REQUEST',     -- customer changed their mind / never showed up
  'OUT_OF_STOCK',         -- kitchen can't fulfill an item
  'KITCHEN_OVERLOADED',   -- merchant cancelling to manage queue
  'PAYMENT_FAILED',       -- (PAY_AFTER) bill couldn't be settled

  'STAFF_ERROR',          -- wrong table / wrong items / misclick

  -- buggy migration), the database refuses duplicate (tenant, day, number).

  CONSTRAINT orders_lifecycle_monotonic

      (preparing_at IS NULL OR submitted_at IS NOT NULL) AND
      (ready_at     IS NULL OR preparing_at IS NOT NULL) AND
      (completed_at IS NULL OR ready_at     IS NOT NULL)

);

CREATE INDEX ON orders (session_id);
CREATE INDEX ON orders (qr_context_id);                                         -- QR-attribution analytics
CREATE INDEX ON orders (tenant_id, created_at DESC);                            -- "today's orders" dashboard hot path

-- The UNIQUE (tenant_id, order_date, order_number) above also serves as the
-- index for "show me LB-042 from 2026-04-22" merchant-portal lookups.
```

## Part 3 — Column-by-Column

### `id` — TEXT PRIMARY KEY

- **Nullable:** No
- **Default:** None (app-generated cuid)
- **Purpose:** Internal unique identifier for this order.
- **Constraints:** Primary key.
- **Why:** System-internal, used for all FK references (`order_items.order_id`, `order_status_history.order_id`, `bill_orders.order_id`, `kitchen_tickets.order_id`). Never exposed to customers -- the customer sees `order_number` on receipts and `order_token` in URLs.

### `tenant_id` — TEXT NOT NULL

- **Nullable:** No
- **Default:** None
- **Purpose:** The tenant (restaurant/stall) that owns this order.
- **Constraints:** `REFERENCES tenants(id) ON DELETE CASCADE`. Indexed as part of `(tenant_id, status)`.
- **Why:** Every order belongs to exactly one tenant. This column is the backbone of tenant isolation. Every query in the application layer includes `WHERE tenant_id = ?`, enforced by the `TenantGuard` middleware reading the JWT. CASCADE ensures tenant deletion removes all orders.

### `session_id` — TEXT

- **Nullable:** Yes
- **Default:** None
- **Purpose:** Links this order to an order session, if one exists.
- **Constraints:** `REFERENCES order_sessions(id)`. Indexed. Cross-FK tenant parity enforced by `orders_session_tenant_parity` trigger.
- **Why:** Nullable because `STALL_KIOSK + PAY_BEFORE` orders have no session. Each order is a standalone transaction. For all other combinations, the order belongs to a session:
  - `STALL_KIOSK + PAY_AFTER`: session groups orders for bill consolidation.
  - `DINE_IN_TABLE + PAY_BEFORE`: session tracks the table occupancy.
  - `DINE_IN_TABLE + PAY_AFTER`: session groups orders for table tracking and bill consolidation.

  The parity trigger prevents cross-tenant contamination: an order for Tenant A cannot reference a session owned by Tenant B.

### `order_date` — DATE NOT NULL

- **Nullable:** No
- **Default:** None (set by the application from the tenant-local date at order creation)
- **Purpose:** The tenant-local calendar date the order was placed on. Pairs with `order_number` to form the `(tenant_id, order_date, order_number)` UNIQUE constraint.
- **Constraints:** Part of the composite UNIQUE constraint `orders_tenant_day_number_unique`.
- **Why:** Two reasons this is its own column rather than `created_at::date`:

  1. **DB-level uniqueness for the daily-reset numbering scheme.** Without this column, "today's `LB-042` and yesterday's `LB-042` are different orders" can only be enforced by the `allocate_order_number()` row-lock — which is correct but relies on every code path going through the allocator. With `(tenant_id, order_date, order_number)` UNIQUE, the database itself refuses duplicates even if a raw-SQL fix, batch import, or buggy migration bypasses the allocator. **Belt-and-braces.**

  2. **Tenant-local, not UTC.** `created_at::date` would be UTC, so a Cambodia merchant ordering at 22:00 ICT (15:00 UTC) on April 22 would have `created_at::date = 2026-04-22 UTC` — fine. But a 23:30 ICT order would have `created_at::date = 2026-04-22 UTC` (16:30 UTC) — also fine. A 06:30 ICT order on the next day would have `created_at::date = 2026-04-22 UTC` again (23:30 UTC the previous day) — **wrong for daily numbering**. Storing `order_date` explicitly (computed from `tenant_settings.timezone`) avoids the timezone trap.

  The value is set by `allocate_order_number(tenant_id)` itself, alongside the number — same transaction, no race window.

### `order_number` — TEXT NOT NULL

- **Nullable:** No
- **Default:** None (generated by `allocate_order_number(tenant_id)`)
- **Purpose:** The human-readable, customer-facing order number displayed on receipts, kitchen tickets, and the status page. Shown to customers, kitchen staff, cashiers, and merchants — the single identifier everyone references.
- **Constraints:** `NOT NULL`. Part of the composite `UNIQUE (tenant_id, order_date, order_number)` (added 2026-04-24).
- **Why:** Formatted as `{tenants.code_prefix}-{daily_counter_zero_padded}`, e.g. `LB-001`, `LB-042`, `LB-999`, growing to `LB-1024` on high-volume days. Zero-padded to 3 digits; grows naturally beyond that with no hard cap. Generated atomically by `allocate_order_number()`:

  ```sql
  SELECT allocate_order_number('tenant_lucky');
  -- Returns: 'LB-042'
  ```

  Key properties:
  - **Tenant-prefixed** — the `code_prefix` ensures cross-tenant uniqueness at a glance (`LB-042` and `BQ-042` are obviously different businesses).
  - **Daily reset** — counter resets to `1` at the tenant's local midnight (from `tenant_settings.timezone`). The kitchen's mental model matches: "042 is today's 42nd order." Yesterday's orders are ancient history.
  - **Short and memorable** — customers can say it aloud ("I'm LB-042"); staff can write it on a sticky note; it fits comfortably on a kitchen tablet header.
  - **Monotonically increasing within a day** — on 2026-04-22 at Lucky Burger, LB-042 came before LB-043. Across days, no ordering semantic.
  - **Unique within a (tenant, day)** — enforced two layers deep: the allocator's row-level lock on `tenant_sequences` prevents same-day allocation collisions, AND the database refuses duplicate `(tenant_id, order_date, order_number)` rows via the `orders_tenant_day_number_unique` constraint (added 2026-04-24). Cross-day reuse is the whole point of daily reset and is permitted by the constraint shape.
  - **Not an access credential** — guessing a display number gets an attacker nothing. URL access uses `order_token`.

  See `design-discussions/order-numbering-strategy.md` for the full design rationale and industry comparison (Gong Cha, Shopify, Toast, etc.).

### `order_token` — TEXT UNIQUE NOT NULL

- **Nullable:** No
- **Default:** None (app-generated, cryptographically random)
- **Purpose:** The public, unguessable identifier for the order status page.
- **Constraints:** `UNIQUE` globally.
- **Why:** The URL `xfos.com/status/{order_token}` is how anonymous customers (no account, no login) check their order status. The token must be:
  - **Unguessable** -- if sequential or predictable, anyone could enumerate all orders for all tenants. This is a privacy and security requirement.
  - **Globally unique** -- the status page URL does not include a tenant slug. The system resolves the tenant from the token alone (like `qr_contexts.token`).
  - **Immutable** -- once generated, the token never changes. Customers may bookmark the URL or share it.

  Typically generated as a 22-character base62 string (alphanumeric, URL-safe, no encoding needed), yielding ~131 bits of entropy.

### `order_token_expires_at` — TIMESTAMP(3) (nullable)

- **Nullable:** Yes (NULL means the token never expires)
- **Default:** `NULL`
- **Purpose:** Optional expiry for the public status-page link. After this time, the storefront returns a "this link has expired" message instead of order details.
- **Constraints:** None.
- **Why:** Privacy and cleanup. Most customers check their status page within minutes or hours of ordering; after the order is `COMPLETED` (or `CANCELLED`), the link has minimal long-term value. A reasonable cleanup policy is to set `order_token_expires_at = completed_at + 90 days` so older order details cannot be enumerated indefinitely from URLs that may have leaked into chat archives, screenshots, or email threads.

  At MVP this column is **nullable and unset by default** — no expiry policy is in force. The column exists so a policy can be turned on later (e.g., per-tenant or platform-wide) without a schema migration. When a token resolution finds a row with `order_token_expires_at < NOW()`, the storefront returns a generic "link expired" page (no tenant info, no order info — same response shape as a not-found token).

  Distinct from token *revocation* (an `order_token_revoked_at` column is reserved for a future need but not added today — YAGNI).

### `status` — "OrderStatus" NOT NULL

- **Nullable:** No
- **Default:** `'SUBMITTED'`
- **Purpose:** The current lifecycle state of the order.
- **Constraints:** Must be one of: `SUBMITTED`, `PREPARING`, `READY`, `COMPLETED`, `CANCELLED`.
- **Why:** The order status enum represents a pure order lifecycle, with no payment concerns mixed in. Payment is tracked entirely by `BillStatus` + `PaymentStatus`. See `design-discussions/order-status-redesign.md` for the full redesign rationale.

  ```
  OrderStatus enum values:
    SUBMITTED   — order exists, kitchen is queued (initial status for all orders)
    PREPARING   — kitchen has started (auto-synced when first ticket moves to PREPARING)
    READY       — all items ready for pickup/serving (auto-synced when all tickets reach READY)
    COMPLETED   — customer received the food (staff confirms pickup/delivery)
    CANCELLED   — order was cancelled (by customer, merchant, or system)
  ```

  **State transitions:**

  ```
  SUBMITTED  → PREPARING    (first kitchen ticket moves to PREPARING)
  SUBMITTED  → CANCELLED    (customer cancels, merchant cancels, system timeout)
  PREPARING  → READY        (all kitchen tickets reach READY)
  PREPARING  → CANCELLED    (merchant cancels — kitchen can't fulfill)
  READY      → COMPLETED    (staff confirms pickup/delivery, or auto-timeout)
  READY      → CANCELLED    (rare — merchant voids after food is ready)
  ```

  **Why the default is `SUBMITTED`:** The order record is only created after the payment gate (for PAY_BEFORE) or at submission time (for PAY_AFTER). In both cases, the order is already accepted the moment it exists. There is no "waiting for payment" state on orders — that concern lives on `BillStatus`.

  **Auto-sync from kitchen tickets:** Kitchen staff only interact with ticket statuses. The backend automatically syncs order status from ticket state changes:
  - First ticket -> PREPARING: order moves to PREPARING
  - All tickets -> READY: order moves to READY
  - Staff confirms pickup: order moves to COMPLETED

  ```sql
  CREATE TYPE "OrderStatus" AS ENUM ('SUBMITTED', 'PREPARING', 'READY', 'COMPLETED', 'CANCELLED');
  ```

### `service_model` — "ServiceModel" NOT NULL

- **Nullable:** No
- **Default:** None (snapshotted from `tenant_settings.service_model` at order creation)
- **Purpose:** Records whether this order was placed at a stall/kiosk or at a numbered table.
- **Constraints:** Must be one of: `STALL_KIOSK`, `DINE_IN_TABLE`.
- **Why:** Snapshotted at order creation time for two reasons:
  1. **The kitchen ticket needs it.** A `DINE_IN_TABLE` ticket shows "Table 5" and implies delivery to the table. A `STALL_KIOSK` ticket shows a pickup number and implies counter pickup. The kitchen display renders differently based on this value.
  2. **Historical accuracy.** If the tenant later changes their service model (e.g., adds tables), existing orders retain their original context.

  ```sql
  CREATE TYPE "ServiceModel" AS ENUM ('STALL_KIOSK', 'DINE_IN_TABLE');
  ```

### `pay_timing` — "PayTiming" NOT NULL

- **Nullable:** No
- **Default:** None (snapshotted from `tenant_settings.pay_timing` at order creation, overridable)
- **Purpose:** Records when payment is expected for this specific order.
- **Constraints:** Must be one of: `PAY_BEFORE`, `PAY_AFTER`.
- **Why:** This is the single most important behavioral column in the system. It determines:
  - **When the order record is created:** `PAY_BEFORE` -> order created after payment succeeds. `PAY_AFTER` -> order created immediately at submission.
  - **Kitchen ticket timing:** `PAY_BEFORE` -> ticket created after payment (same moment as order). `PAY_AFTER` -> ticket created immediately (same moment as order).
  - **Bill grouping:** `PAY_BEFORE` -> one bill per order (created before the order). `PAY_AFTER` -> one bill per session (multiple orders, created when customer requests it).
  
  In both cases, the initial order status is always `SUBMITTED`. The `pay_timing` column no longer determines the initial status — it determines when the order is born relative to payment.

  **Why snapshotted per-order, not read from tenant_settings at query time:**
  - The merchant may change `tenant_settings.pay_timing` at any time. Existing orders must retain their original behavior.
  - Per-order overrides are possible: a merchant can grant pay-after treatment to a regular customer at a normally pay-before stall. The override is stored here, not on `tenant_settings`.
  - The use-case layer reads `orders.pay_timing`, never `tenant_settings.pay_timing`, to determine order behavior. This eliminates a join and a potential race condition.

  ```sql
  CREATE TYPE "PayTiming" AS ENUM ('PAY_BEFORE', 'PAY_AFTER');
  ```

  See `design-discussions/servicemodel-and-paytiming.md` for the complete design decision, all four scenario walkthroughs, and the rationale for simplifying from 3+3 enums to 2+2.

### `table_ref` — TEXT

- **Nullable:** Yes
- **Default:** None
- **Purpose:** The human-readable table identifier for this order, if applicable.
- **Constraints:** None.
- **Why:** Denormalized from `order_sessions.table_ref` (which itself was denormalized from `qr_contexts.table_ref`). NULL for `STALL_KIOSK` orders. Set for `DINE_IN_TABLE` orders.

  Denormalized because:
  - Kitchen tickets read `table_ref` from the order, not by joining to the session. This is a hot-path read on every ticket render.
  - The value must survive session closure, QR relabeling, or session deletion. The order's `table_ref` is a historical snapshot.

### `subtotal_cents` — INTEGER NOT NULL

- **Nullable:** No
- **Default:** None (calculated by the application as `SUM(order_items.line_total_cents)`)
- **Purpose:** The sum of all line item totals before any discounts, taxes, or service charges.
- **Constraints:** `CHECK (subtotal_cents >= 0)` (H5).
- **Why:** Stored as integer cents to avoid floating-point errors. $12.50 is stored as `1250`. At MVP, `subtotal_cents = total_cents` because there are no discounts, taxes, or service charges. The columns are separated now so that adding those features later does not require a schema change.

### `total_cents` — INTEGER NOT NULL

- **Nullable:** No
- **Default:** None (calculated by the application)
- **Purpose:** The final amount the customer pays for this order.
- **Constraints:** `CHECK (total_cents >= 0)` (H5). `CHECK (total_cents >= subtotal_cents)` (H5).
- **Why:** At MVP, `total_cents = subtotal_cents`. The `total_cents >= subtotal_cents` constraint ensures that discounts or adjustments cannot make the total negative or less than the subtotal. Future features (discounts, taxes, rounding) modify `total_cents` relative to `subtotal_cents`.

  Stored as integer cents. $46.00 is `4600`.

### `currency` — "Currency" NOT NULL

- **Nullable:** No
- **Default:** `'USD'`
- **Purpose:** The ISO 4217 currency code for all monetary values on this order.
- **Constraints:** Postgres `"Currency"` enum (`USD`, `KHR`).
- **Why:** Cambodia operates primarily in USD for everyday transactions, with KHR (Cambodian Riel) as an alternative. The currency is snapshotted per-order because:
  - A tenant might change their currency setting (unlikely but possible).
  - Historical orders must record what currency was used at the time of the transaction.
  - Billing, reporting, and payment gateway integration all need to know the currency.

  Default is `'USD'` matching `tenant_settings.currency` default.

### `notes` — TEXT

- **Nullable:** Yes
- **Default:** None
- **Purpose:** Order-level special instructions from the customer.
- **Constraints:** None.
- **Why:** Distinct from `order_items.notes` (per-item instructions). Order-level notes are for things like: "Please deliver all dishes at the same time", "Allergic to peanuts -- please check all items", or "Call 012-xxx-xxxx when ready." In Khmer: "សូមយកមកព្រមគ្នា" ("Please bring together").

### `version` — INTEGER NOT NULL DEFAULT 1

- **Nullable:** No
- **Default:** `1`
- **Purpose:** Optimistic-concurrency counter. Every UPDATE on this row must include `WHERE version = $expected` and bump `version + 1`. Prevents lost updates when multiple actors modify the same order simultaneously.
- **Constraints:** None at the DB level — enforced by the application layer.
- **Why:** Order rows take updates from several concurrent paths: (1) kitchen-ticket auto-sync writing `status` transitions, (2) merchant-portal staff manually completing or cancelling, (3) bill/payment flows updating `pay_timing` overrides or finalising amounts. Two of these can land in the same millisecond — staff taps "Complete" while the ticket sync also tries to fire. Without `version`, the second update silently overwrites the first (e.g., a manual cancel reverts because the ticket sync wrote AFTER it). With `version`, the loser sees `0 rows affected`, refetches state, and retries cleanly. Same OCC pattern as `carts.version`.

### `cancellation_reason` — "OrderCancellationReason" (nullable)

- **Nullable:** Yes
- **Default:** `NULL`
- **Purpose:** When `status = 'CANCELLED'`, records *why* the order was cancelled. NULL otherwise.
- **Constraints:** `CHECK ((status = 'CANCELLED') OR (cancellation_reason IS NULL))` — guarantees the column is meaningful only on cancelled rows.
- **Why:** Cancellations are operational events that demand investigation when they spike. Without segmentation, "12 cancellations today" is an alarm with no signal. The reason column lets dashboards distinguish "kitchen ran out of an ingredient" (`OUT_OF_STOCK`) from "we have a walkaway problem" (`CUSTOMER_REQUEST` clusters from anonymous customers) from "staff training issue" (`STAFF_ERROR` clusters by `cancelled_by_id`). Splitting the lifecycle enum (e.g., `CANCELLED_BY_CUSTOMER`, `CANCELLED_BY_KITCHEN`) was rejected — it would force every status check to enumerate variants forever. See [`enums/order-cancellation-reason.md`](../enums/order-cancellation-reason.md).

### `cancelled_by_id` — TEXT (nullable)

- **Nullable:** Yes
- **Default:** `NULL`
- **Purpose:** When a staff member cancels an order, records *who*. NULL for system-driven cancellations (`PAYMENT_FAILED`, `DUPLICATE`, `SYSTEM_TIMEOUT`) and for cancellations the customer self-services through the storefront.
- **Constraints:** `REFERENCES users(id)`. `CHECK ((cancelled_by_id IS NULL) OR (status = 'CANCELLED'))` — set only when the row is cancelled.
- **Why:** Accountability and dispute resolution. The `_id` suffix follows the schema's existing FK-naming convention (`tenant_id`, `closed_by_id` on carts, etc.).

### `created_by_id` — TEXT (nullable)

- **Nullable:** Yes (required when `source = 'MERCHANT_MANUAL'`)
- **Default:** `NULL`
- **Purpose:** Staff member who entered the order on behalf of the customer. NULL for self-service orders.
- **Constraints:** `REFERENCES users(id)`. Two CHECKs:
  - `orders_storefront_has_no_creator` — `STOREFRONT_QR` orders forbid `created_by_id` (a customer scanning the QR cannot be a `users` row at MVP).
  - `orders_manual_has_creator` — `MERCHANT_MANUAL` orders require `created_by_id` (no anonymous staff entries).
- **Why:** Two real MVP use cases:
  1. **Walk-up at a stall** — a customer arrives at a noodle stall and orders directly to the cashier; the cashier enters the order in the merchant portal.
  2. **Elderly dine-in customer** — a customer at a sit-down restaurant asks the waiter to enter their order from a tablet because the QR-scan flow is too unfamiliar.

  In both cases, the system must record which staff member entered the order — for accountability, training feedback ("which staff make the most mistakes?"), and dispute resolution.

### `estimated_ready_at` — TIMESTAMP(3) (nullable)

- **Nullable:** Yes
- **Default:** `NULL` (set by the kitchen-ticket service when the ticket is dispatched)
- **Purpose:** The system's best estimate of when the order will be ready for pickup or service.
- **Constraints:** None.
- **Why:** Customer-facing ETA on the status page — "Ready in ~10 min" rather than just progress dots. Computed at ticket-dispatch time as `submitted_at + average_prep_time_for_items`. As the kitchen progresses, the estimate may be refined (or simply replaced by `ready_at` when the food is actually ready). This is product-essential UX, not enterprise window dressing — customers want to know when to come back.

  When NULL: the order has just been submitted and the estimate hasn't been computed yet, OR the order was created from a flow that doesn't compute an estimate (e.g., merchant-manual entry).

### Lifecycle transition timestamps

`order_status_history` records the full audit trail of every transition. These columns denormalize the most-asked-about transition timestamps onto the row itself so analytics queries don't require window functions over the history table.

#### `submitted_at` — TIMESTAMP(3)

- **Nullable:** Yes
- **Default:** None
- **Purpose:** When the order was submitted to the kitchen (status = `SUBMITTED`, which is the initial status).
- **Constraints:** None.
- **Why:** For both `PAY_BEFORE` and `PAY_AFTER`, `submitted_at` is approximately equal to `created_at` because the order record is only created when submission actually happens. The distinction between the two timestamps is:
  - `created_at` = when the database row was inserted (Postgres `CURRENT_TIMESTAMP`).
  - `submitted_at` = when the application confirmed kitchen submission (set by application code).

  The gap between the two is typically milliseconds (within the same transaction). The more useful metric is the gap between `submitted_at` and subsequent status changes (PREPARING, READY) — measuring kitchen response time and preparation time.

  NULL for orders that are cancelled before the kitchen starts (edge case: order cancelled in the brief window between creation and ticket dispatch).

#### `preparing_at` — TIMESTAMP(3) (nullable)

- **Nullable:** Yes (set when status transitions to `PREPARING`)
- **Default:** None
- **Purpose:** When kitchen actually started cooking — the first ticket reached `PREPARING`.
- **Constraints:** None.
- **Why:** Without this column, "average kitchen response time" requires `MIN(occurred_at) FROM order_status_history WHERE status='PREPARING'` — a window function joined to orders. With it, the merchant dashboard query is `AVG(preparing_at - submitted_at)` — an indexable scan over orders.

#### `ready_at` — TIMESTAMP(3) (nullable)

- **Nullable:** Yes (set when all kitchen tickets reach `READY`)
- **Default:** None
- **Purpose:** When the food was ready for pickup or service.
- **Constraints:** None.
- **Why:** Drives the most operationally important metric: actual prep time = `ready_at - preparing_at`. Also enables auto-completion timeouts ("if not picked up within 15 min of ready, escalate").

#### `completed_at` — TIMESTAMP(3) (nullable)

- **Nullable:** Yes (set when staff confirms pickup/delivery)
- **Default:** None
- **Purpose:** When the order was delivered to the customer.
- **Constraints:** None.
- **Why:** Total customer wait = `completed_at - submitted_at`. A core KPI for tenant operations and platform-wide health.

#### `cancelled_at` — TIMESTAMP(3) (nullable)

- **Nullable:** Yes (set when status transitions to `CANCELLED`)
- **Default:** None
- **Purpose:** When the order was cancelled.
- **Constraints:** None.
- **Why:** Pairs with `cancellation_reason` and `cancelled_by_id` for cancellation analytics: "what time of day do most cancellations happen?", "are cancellations clustering during shift handovers?"

### Monetary breakdown

These columns are added now so the addition of discounts, taxes, and service charges later is **not a schema change** — only an app-layer pricing-engine change.

#### `discount_cents` — INTEGER NOT NULL DEFAULT 0

- **Nullable:** No
- **Default:** `0`
- **Purpose:** Total discounts applied to this order (promo codes, loyalty rewards, manual percent-off, etc.).
- **Constraints:** `CHECK (discount_cents >= 0)` and `CHECK (discount_cents <= subtotal_cents)`.
- **Why:** Stored as a positive number (subtracted from subtotal in the formula). Cannot exceed subtotal — the platform does not support negative pricing. At MVP this is always `0`.

#### `tax_cents` — INTEGER NOT NULL DEFAULT 0

- **Nullable:** No
- **Default:** `0`
- **Purpose:** Total tax applied (VAT or local sales tax).
- **Constraints:** `CHECK (tax_cents >= 0)`.
- **Why:** Cambodia has VAT applicable to some establishments (10% for VAT-registered businesses). At MVP most small stalls are below the VAT threshold so this is `0`; the column lets larger tenants add VAT without a migration.

#### `service_charge_cents` — INTEGER NOT NULL DEFAULT 0

- **Nullable:** No
- **Default:** `0`
- **Purpose:** Service charge or tip (typically 5–10% at sit-down restaurants).
- **Constraints:** `CHECK (service_charge_cents >= 0)`.
- **Why:** Common at dine-in restaurants in Cambodia. Separated from `tax_cents` so reporting can distinguish "money the merchant keeps" from "money the merchant remits to the government."

#### Total formula CHECK

```sql
CHECK (total_cents = subtotal_cents - discount_cents + tax_cents + service_charge_cents)
```

Replaces the old `total >= subtotal` check. The DB now enforces the exact pricing math — the application can never write an inconsistent total even by mistake.

### `qr_context_id` — TEXT (nullable)

- **Nullable:** Yes
- **Default:** `NULL`
- **Purpose:** The QR code that initiated this order, snapshotted at order creation.
- **Constraints:** `REFERENCES qr_contexts(id)`. Indexed. Cross-FK tenant parity enforced by `orders_qr_tenant_parity` trigger.
- **Why:** QR-attribution analytics. Two real questions need this column:
  1. **"Which QR codes drive the most revenue?"** — for merchants with many QR placements (counter, takeaway, multiple tables), this is essential.
  2. **`STALL_KIOSK + PAY_BEFORE` orders have no session.** Without this column, the QR origin is lost entirely for the highest-volume order flow. With it, attribution works end-to-end across all flows.

  When NULL: the order was not initiated by a QR scan (e.g., `source = 'MERCHANT_MANUAL'`).

### `source` — "OrderSource" NOT NULL DEFAULT 'STOREFRONT_QR'

- **Nullable:** No
- **Default:** `'STOREFRONT_QR'`
- **Purpose:** Records the channel through which the order was created.
- **Constraints:** Pairs with `created_by_id` via two CHECKs (see `created_by_id`).
- **Why:** Distinguishes customer self-service from staff-entered orders, and forward-compatible with future channels:
  - `STOREFRONT_QR` — customer scanned a QR and ordered through the storefront. Default and most common.
  - `MERCHANT_MANUAL` — staff entered the order in the merchant portal (walk-up at stall; elderly customer at dine-in restaurant; phone-in pickup).
  - `API` — (future) third-party integration (delivery aggregator, partner POS).
  - `MOBILE_APP` — (future) Phase 2 customer mobile app for marketplace and pickup.

  See [`enums/order-source.md`](../enums/order-source.md).

### `created_at` — TIMESTAMP(3) NOT NULL

- **Nullable:** No
- **Default:** `CURRENT_TIMESTAMP`
- **Purpose:** When the order record was created in the database.
- **Constraints:** None.
- **Why:** For both `PAY_BEFORE` and `PAY_AFTER`, this marks the moment the order was accepted and sent to the kitchen. For `PAY_AFTER`, this is when the customer tapped "Place Order." For `PAY_BEFORE`, this is when the payment succeeded and the system created the order — the customer's original tap happened earlier (the time between tap and payment is measured on the bill/payment side, not the order side).

### `updated_at` — TIMESTAMP(3) NOT NULL

- **Nullable:** No
- **Default:** Maintained by Prisma (`@updatedAt`).
- **Purpose:** When this order record was last modified (status change, notes update, etc.).
- **Constraints:** None.
- **Why:** Standard Prisma convention. Useful for optimistic concurrency checks and for sorting "recently updated" views in the merchant portal.

## Part 4 — Indexes

### `PRIMARY KEY (id)`

- **What it serves:** All internal lookups and FK joins.
- **Example:** `SELECT * FROM orders WHERE id = 'order_001'`

### `UNIQUE (tenant_id, order_date, order_number)` — composite

- **What it serves:** Two things at once.
  1. **Belt-and-braces uniqueness.** Even if the allocator is bypassed (raw SQL fix, batch import, buggy migration), the database refuses duplicate `(tenant, day, number)` rows. The application layer no longer carries the entire correctness burden alone.
  2. **Merchant-portal lookup by order number.** "Show me LB-042 from 2026-04-22" is now a direct index hit:
     ```sql
     SELECT * FROM orders
     WHERE tenant_id   = 'tenant_lucky'
       AND order_date   = '2026-04-22'
       AND order_number = 'LB-042';
     ```
- **Why composite, not just `(tenant_id, order_number)`:** With daily reset, `LB-042` recurs every day; a `(tenant_id, order_number)` UNIQUE would reject yesterday's collision incorrectly. Including `order_date` makes the constraint exactly express "one of each number per tenant per day," which is the actual rule.
- **For cross-day queries** (audit, support), use `id` (cuid), `order_token`, or scan with `created_at` ranges.

### `UNIQUE (order_token)`

- **What it serves:** The public status page lookup. This is the highest-traffic customer-facing query.
- **Example:**
  ```sql
  SELECT o.id, o.tenant_id, o.order_number, o.status,
         o.total_cents, o.currency, o.table_ref, o.service_model
  FROM orders o
  WHERE o.order_token = 'abc123def456xyz789';
  ```
- **Why globally unique:** The status page URL `xfos.com/status/{order_token}` does not include a tenant identifier. The token must resolve to exactly one order across all tenants.

### `INDEX ON orders (tenant_id, status)`

- **What it serves:** The merchant portal's order dashboard -- the most common merchant-facing query.
- **Example (active orders):**
  ```sql
  SELECT o.id, o.order_number, o.status, o.total_cents,
         o.table_ref, o.created_at
  FROM orders o
  WHERE o.tenant_id = 'tenant_abc'
    AND o.status IN ('SUBMITTED', 'PREPARING', 'READY')
  ORDER BY o.created_at DESC;
  ```
- **Example (daily sales):**
  ```sql
  SELECT COUNT(*), SUM(total_cents) AS revenue_cents
  FROM orders
  WHERE tenant_id = 'tenant_abc'
    AND status NOT IN ('CANCELLED')
    AND created_at >= '2026-04-09 00:00:00'
    AND created_at < '2026-04-10 00:00:00';
  ```
- **Why:** Every merchant portal query filters by `tenant_id` first, then by `status`. The composite index serves both in one B-tree traversal.

### `INDEX ON orders (session_id)`

- **What it serves:** Session-to-orders lookup. Used when generating a bill ("find all orders in this session") and when the storefront shows the running order list for a session.
- **Example:**
  ```sql
  SELECT id, order_number, status, total_cents
  FROM orders
  WHERE session_id = 'sess_001'
    AND status != 'CANCELLED'
  ORDER BY created_at ASC;
  ```
- **Why a separate index:** This query does not filter by `tenant_id` first (the session ID already implies the tenant). A composite `(tenant_id, session_id)` would also work but is redundant given that `session_id` is globally unique.

## Part 7 — Design Decisions

### Why `pay_timing` is per-order, not only on `tenant_settings`

Three options were considered:
1. **Tenant-level only** -- read from `tenant_settings` at query time.
2. **Tenant-level default, snapshotted on order** (chosen).
3. **Customer chooses** at checkout.

Option 1 was rejected because changing `tenant_settings.pay_timing` would retroactively change the behavior of in-flight orders. Option 3 was rejected because letting anonymous customers choose "pay later" creates credit risk and adds cognitive load. Option 2 provides the flexibility of per-order overrides while keeping the default simple.

Full rationale in `design-discussions/servicemodel-and-paytiming.md`, Decision 3.

### Why `order_number` is Postgres-generated, not Redis

The original design used Redis `INCR` for order number generation. This was replaced with the `allocate_order_number()` Postgres function (C3) for three reasons:
1. **Atomicity** -- the counter increment and order INSERT happen in the same transaction. If the INSERT fails, the counter is not incremented. Redis `INCR` is fire-and-forget and would create gaps.
2. **No external dependency** -- Redis outage would block order creation. Postgres is already a hard dependency.
3. **No split-brain** -- with Redis, a network partition could cause duplicate numbers. With Postgres, the row lock prevents concurrent allocation of the same number.

### Why `order_token` is globally unique

The status page URL `xfos.com/status/{order_token}` does not include a tenant slug. This is intentional:
- Shorter URLs fit better in SMS and messaging apps (common in Cambodia).
- No tenant information is leaked in the URL.
- The customer does not need to know (or type) the tenant name to check their order.

### Why `status` defaults to `SUBMITTED`

The order record is only created after the payment gate has been satisfied (for PAY_BEFORE) or at submission time (for PAY_AFTER). In both cases, the order is already accepted the moment it exists. There is no "waiting for payment" state on orders — that concern is handled entirely by `BillStatus` (OPEN -> PAID) and `PaymentStatus` (INITIATED -> PENDING -> SUCCEEDED). The order status tracks only the kitchen lifecycle: `SUBMITTED -> PREPARING -> READY -> COMPLETED`. See `design-discussions/order-status-redesign.md` for the full rationale behind removing `PENDING_PAYMENT` and `CONFIRMED` from the order status enum.

### Why monetary values are INTEGER cents, not DECIMAL

Floating-point arithmetic is unsafe for money (`0.1 + 0.2 != 0.3` in IEEE 754). Integer cents eliminate this entire class of bug. $3.50 is `350`. All arithmetic is integer addition and multiplication. Display formatting (adding the decimal point) is a presentation concern, not a storage concern.

### Why `subtotal_cents` and `total_cents` are both stored (seemingly redundant at MVP)

At MVP, `total_cents = subtotal_cents` because there are no discounts, taxes, or service charges. But the columns are separated now to avoid a schema migration later. When discounts or taxes are added:
- `subtotal_cents` = sum of line items (unchanged).
- `total_cents` = `subtotal_cents - discount + tax + service_charge`.
- The CHECK constraint `total_cents >= subtotal_cents` will need to be adjusted to allow discounts.

### Why `table_ref` is denormalized from sessions

The kitchen ticket for a `DINE_IN_TABLE` order shows "Table 5." This value is read from `orders.table_ref`, not by joining to `order_sessions`. Reasons:
- Hot-path performance (kitchen display polls frequently).
- Historical accuracy (the session might close, the QR might be relabeled).
- The order is the source of truth for "where was this food supposed to go."

### Why `order_date` is its own column (not derived from `created_at`)

Two reasons:

1. **DB-level uniqueness needs a discrete value.** `UNIQUE (tenant_id, order_date, order_number)` requires `order_date` to be a column the database can index. PostgreSQL does support unique indexes on expressions (`UNIQUE (tenant_id, (created_at AT TIME ZONE timezone)::date, order_number)`), but the expression would need the tenant's timezone — which lives on `tenant_settings`. Cross-table expression indexes are clumsy. A plain DATE column is simpler and faster.

2. **Tenant-local, not UTC.** A Cambodia merchant ordering at 06:30 ICT on 2026-04-23 has `created_at = 2026-04-22 23:30 UTC` — the UTC date is *yesterday*. Daily numbering must be tenant-local; the column captures that explicitly.

The `allocate_order_number(tenant_id)` function sets `order_date` and `order_number` together in the same transaction, so the pairing is atomic.

### Why `order_token_expires_at` exists (and `order_token_revoked_at` does not — yet)

Status-page URLs may live in chat archives, screenshots, and email threads for years. After the order is `COMPLETED`, the customer almost never re-checks. A nullable expiry column lets a future cleanup policy (e.g., "expire 90 days after completion") run without a schema migration. NULL = no expiry, which is the MVP behaviour.

Token *revocation* (a separate `order_token_revoked_at`) was considered and deferred — there is no current product flow that revokes a single order's URL. If a real request lands ("customer asked us to make their old order unfindable"), it can be added as another nullable TIMESTAMP(3) without breaking anything.

### Why a monotonic-lifecycle CHECK

`preparing_at` cannot exist without `submitted_at`; `ready_at` cannot exist without `preparing_at`; `completed_at` cannot exist without `ready_at`. This is a property of the lifecycle. Encoding it in the DB as a single `CHECK` makes a whole class of bugs in the status-sync handler impossible:

- A racy ticket-sync writing `ready_at` without first setting `preparing_at` → rejected.
- A staff "Complete" tap on an order that never reached `READY` → rejected.
- A migration filling in only some of the timestamps for backfilled rows → rejected.

The trade-off is a tiny correctness gain on every UPDATE; in practice it's negligible. Cancellation (`cancelled_at`) sits outside the chain — an order can be cancelled from any status, so it has no monotonic predecessor in the CHECK.

### Why `STOREFRONT_QR` requires `qr_context_id`

The three CHECKs around `source` now form a complete logic table:

| `source` | `created_by_id` | `qr_context_id` |
|---|---|---|
| `STOREFRONT_QR` | **must be NULL** | **must be NOT NULL** |
| `MERCHANT_MANUAL` | **must be NOT NULL** | optional |
| `API` / `MOBILE_APP` | optional | optional |

`STOREFRONT_QR` literally means "a customer scanned a QR code" — if there's no QR context attached, the source is a lie. This CHECK closes a hole in the previous constraint set where you could write `source = 'STOREFRONT_QR'` with `qr_context_id = NULL` and the database would accept it.

### Why `version` (optimistic concurrency) on orders

Order rows are written by several concurrent paths: kitchen-ticket auto-sync, merchant-portal staff actions, and bill/payment finalisation. Two of these can land in the same millisecond — staff taps "Complete" while the ticket sync also fires. Without `version`, last-write-wins quietly corrupts state (e.g., a manual cancellation reverts because a stale ticket-sync update wrote AFTER it). Same OCC pattern as `carts.version`. Cheaper than holding row locks across HTTP requests and works correctly across separate transactions.

### Why a separate `cancellation_reason` enum, not more `OrderStatus` values

Splitting the lifecycle enum (e.g., `CANCELLED_BY_CUSTOMER`, `CANCELLED_BY_KITCHEN`, `CANCELLED_BY_PAYMENT`) would conflate two concepts:

- **Lifecycle state** (am I in flight? Did I complete?) belongs in `status`.
- **Operational metadata** (why did the lifecycle end?) belongs in `cancellation_reason`.

A split enum forces every status check (`WHERE status IN ('CANCELLED_BY_X', 'CANCELLED_BY_Y', …)`) to enumerate all variants forever. The sibling-column approach keeps the lifecycle enum stable and makes adding a new reason a non-breaking enum addition. Same design as carts (`abandoned_reason`).

### Why `created_by_id` is required only for `MERCHANT_MANUAL`

The CHECK constraint `orders_manual_has_creator` enforces that staff-entered orders must record the staff member; `orders_storefront_has_no_creator` forbids the column on customer-self-service orders (since at MVP customers are anonymous and have no `users` row). This makes accountability mandatory exactly where it matters and avoids polluting the column for flows that have no meaningful "creator." The future `API` and `MOBILE_APP` sources are flexible — `created_by_id` is optional there pending the integration design.

### Why dedicated transition timestamps instead of querying `order_status_history`

`order_status_history` is the audit log — every transition is recorded there with timestamp and actor. The transition columns (`preparing_at`, `ready_at`, `completed_at`, `cancelled_at`) are denormalized aggregates of "when did this row first reach status X." Reasons:

1. **Analytics queries are 5x simpler.** "Average prep time today" is `AVG(ready_at - preparing_at)` instead of a window function joined to history.
2. **The data is small and never changes.** A status transition fires exactly once per direction; the column is set once and never updated. Storage cost is negligible.
3. **The single-write rule prevents drift.** The status-transition handler writes both the column and the history row in the same transaction; either both succeed or both roll back. No reconciliation logic needed.

### Why a `qr_context_id` on orders even though sessions also have one

Two reasons to denormalize:

1. **`STALL_KIOSK + PAY_BEFORE` orders have no session.** Without `orders.qr_context_id`, the QR origin is lost for the highest-volume order flow. With it, attribution analytics work end-to-end.
2. **Hot-path attribution queries skip the join.** "Top 10 QR codes by revenue today" runs directly on `orders` instead of joining through `order_sessions`.

The cross-FK tenant parity trigger (`orders_qr_tenant_parity`) prevents an order from referencing a QR owned by a different tenant.

### Why monetary breakdown columns now (with default 0)

`subtotal_cents`, `discount_cents`, `tax_cents`, `service_charge_cents`, `total_cents` together encode the full pricing math. At MVP, the discount/tax/service-charge fields are usually `0`, but they enable real merchant requests today (the elderly-dine-in scenario above already uses `service_charge_cents`). Adding them now — instead of when the first feature request lands — avoids a migration on a hot table later. The CHECK formula (`total = subtotal - discount + tax + service_charge`) makes inconsistent totals impossible.

### Why `service_model` is snapshotted on orders

If a tenant transitions from `STALL_KIOSK` to `DINE_IN_TABLE` (e.g., they add tables to their stall), existing orders retain their original `service_model`. The kitchen ticket rendering logic reads `orders.service_model`, not `tenant_settings.service_model`, so old tickets display correctly.
