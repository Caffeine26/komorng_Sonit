# XFOS — PostgreSQL Schema

Generated from `xfos/database/prisma/schema.prisma`.
**Authoritative source remains the Prisma schema.** This document is a
human-readable reference — do not hand-write migrations from it.

Conventions:
- IDs: `TEXT` (cuid generated by app) unless otherwise noted.
- Money: `INTEGER` cents (never float/numeric).
- Currency: `"Currency"` enum ISO 4217 (`USD`, `KHR`), default `'USD'`.
- Locale: `"Locale"` enum (`en`, `km`). English is the primary **system**
  language (admin, internal tooling); customer-facing storefront default
  is `'km'` per tenant.
- Timestamps: `TIMESTAMP(3)` — `createdAt` defaults to `now()`, `updatedAt` is
  maintained by Prisma.
- Tenant-scoped tables: `PRIMARY KEY (tenant_id, id)` + every cross-tenant FK is
  composite (`FOREIGN KEY (tenant_id, x_id) REFERENCES x(tenant_id, id)`). This
  makes cross-tenant linking impossible by construction — no parity triggers.
  Globally-scoped tables (`tenants`, `users`, `user_auth_providers`,
  `refresh_tokens`, `phone_otp_attempts`, `plans`, `plan_features`,
  `subscriptions`) keep single-column `id` PKs. Junction tables with nullable
  tenant_id (`user_roles`, `audit_logs`) keep single-column `id` PKs.
- All enums are Postgres `TYPE` enums; names match the Prisma enum names.

## TENANT — the crown jewel

### `tenants`

The minimal, stable record for a business. Identity only — no contact info,
no address, no branding. Everything operational lives in `tenant_settings`.

Design notes (see `xfos/database/discussion_and_decision.md`):
- `id` is a cuid (TEXT). System-internal; never exposed in URLs.
- `slug` is the human-readable URL key (`xfos.com/t/acme-burger`). Also acts
  as the "tenant code" — no separate `tenant_key` column. Unique per platform.
- Display name is bilingual: `name_en` is required (English is the primary
  system language), `name_km` is optional. Two columns instead of a
  translation table because each tenant has exactly one name in at most
  two locales.
- Owner details (email, phone, personal name) do NOT live here — they belong
  to `users` + `user_roles`. A tenant can have multiple owners, and owners
  come and go; the tenant record outlives them.

```sql
CREATE TABLE tenants (
  id          TEXT PRIMARY KEY,
  slug        TEXT UNIQUE NOT NULL,                           -- URL key (long kebab-case)
  code_prefix TEXT UNIQUE NOT NULL                            -- SHORT prefix for order/bill numbers ('LB', 'BQ')
              CHECK (code_prefix ~ '^[A-Z]{2,4}$'),
  name_en     TEXT NOT NULL,                                  -- English display name (primary, system default)
  name_km     TEXT,                                           -- Khmer display name (optional)
  status      "TenantStatus" NOT NULL DEFAULT 'DRAFT',
  created_at  TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at  TIMESTAMP(3) NOT NULL
);
```

`code_prefix` is globally unique across the platform, immutable, and 2-4
uppercase Latin letters. It is the prefix on every customer-facing order
number (`LB-042`) and bill number (`LB-B-000125`). See
`design-discussions/order-numbering-strategy.md`.

### `tenant_settings`

One row per tenant (enforced by `UNIQUE(tenant_id)`). The configuration
center for each restaurant or stall. See
`design-discussions/tenant-settings-improvements.md` for the full design discussion.

Design notes:
- **JSONB for flexible/bilingual data.** `address`, `description`,
  `business_contacts`, and `social_links` use JSONB — extensible to new
  languages and channels without migrations.
- **Payment methods as enum array.** `"PaymentMethod"[]` replaces 3 booleans —
  scales with new methods without adding columns.
- **Tax as 2 fields.** `tax_rate_bps = 0` means no tax. No separate boolean.
- **GPS as separate columns** (not in JSONB) — needed for numeric distance
  calculations in future delivery/marketplace.
- **Operating hours in a separate table** (`tenant_operating_hours`) — supports
  multiple time slots per day (lunch breaks).

```sql
CREATE TABLE tenant_settings (
  tenant_id       TEXT NOT NULL,
  id              TEXT NOT NULL,

  -- Operational
  service_model        "ServiceModel" NOT NULL DEFAULT 'STALL_KIOSK',
  pay_timing           "PayTiming"    NOT NULL DEFAULT 'PAY_BEFORE',
  default_locale       "Locale" NOT NULL DEFAULT 'km',
  timezone             TEXT NOT NULL DEFAULT 'Asia/Phnom_Penh',
  currency             "Currency" NOT NULL DEFAULT 'USD',
  auto_accept_orders   BOOLEAN NOT NULL DEFAULT TRUE,       -- FALSE = merchant reviews before kitchen

  -- Tax
  tax_rate_bps     INTEGER NOT NULL DEFAULT 0,              -- 0 = no tax. 1000 = 10.00% VAT
  tax_inclusive    BOOLEAN NOT NULL DEFAULT TRUE,            -- price includes tax (Cambodia norm)

  -- Business contacts (multi-channel, multi-value)
  business_contacts  JSONB NOT NULL DEFAULT '[]',           -- [{"type":"phone","value":"+855...","label":"Smart"}, ...]

  -- Store description (bilingual)
  description  JSONB,                                       -- {"en": "Best noodles in BKK1", "km": "មីឆាឆ្ងាញ់បំផុត"}

  -- Address (bilingual structured — filterable by sangkat/khan/province)
  address  JSONB,                                           -- {"street":{"en":"...","km":"..."},"sangkat":{"en":"...","km":"..."},"khan":{"en":"...","km":"..."},"province":{"en":"...","km":"..."},"country":"KH"}

  -- GPS (separate columns for numeric math — future delivery/marketplace)
  latitude        DECIMAL(10, 7),
  longitude       DECIMAL(10, 7),
  google_maps_url TEXT,

  -- Branding
  primary_color     TEXT,
  logo_url          TEXT,                                   -- CDN URL, uploaded via merchant portal
  cover_image_url   TEXT,                                   -- CDN URL, storefront hero banner

  -- Social links
  social_links  JSONB,                                      -- {"facebook":"...","instagram":"...","tiktok":"..."}

  updated_at    TIMESTAMP(3) NOT NULL,

  PRIMARY KEY (tenant_id, id),
  FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON DELETE CASCADE,
  CONSTRAINT tenant_settings_one_per_tenant UNIQUE (tenant_id)
);
```

### `setup_progress`

Onboarding checklist — one row per tenant. Monotonic timestamps for each
milestone, plus a `GENERATED STORED` `go_live_ready` gate. See
`tables/setup-progress.md` for full design.

```sql
CREATE TABLE setup_progress (
  tenant_id                  TEXT NOT NULL,
  id                         TEXT NOT NULL,

  -- Milestone timestamps. NULL = not yet completed. Monotonic.
  profile_completed_at       TIMESTAMP(3),
  menu_completed_at          TIMESTAMP(3),
  translations_completed_at  TIMESTAMP(3),
  payments_configured_at     TIMESTAMP(3),
  qr_created_at              TIMESTAMP(3),

  -- Lifetime bookends
  created_at                 TIMESTAMP(3) NOT NULL DEFAULT NOW(),
  went_live_at               TIMESTAMP(3),
  updated_at                 TIMESTAMP(3) NOT NULL,

  -- Derived gate. Cannot drift from inputs.
  go_live_ready BOOLEAN NOT NULL GENERATED ALWAYS AS (
        profile_completed_at      IS NOT NULL
    AND menu_completed_at         IS NOT NULL
    AND translations_completed_at IS NOT NULL
    AND payments_configured_at    IS NOT NULL
    AND qr_created_at             IS NOT NULL
  ) STORED,

  PRIMARY KEY (tenant_id, id),
  FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON DELETE CASCADE,
  CONSTRAINT setup_progress_one_per_tenant UNIQUE (tenant_id)
);

CREATE INDEX idx_setup_progress_stuck
  ON setup_progress (created_at)
  WHERE go_live_ready = FALSE;
```

### `plans`

Platform-level catalog of subscription tiers. Bilingual display (en + km),
multi-currency, two lifecycle flags (`is_active` = accepting subscriptions,
`is_public` = visible in signup UI). `code` is TEXT (not enum) so adding or
retiring plans requires no migration. See `tables/plans.md`.

```sql
CREATE TABLE plans (
  id                  TEXT PRIMARY KEY,
  code                TEXT UNIQUE NOT NULL,             -- 'STARTER' | 'GROWTH' | 'PRO' (stable machine key)

  -- Bilingual display
  name_en             TEXT NOT NULL,
  name_km             TEXT NOT NULL,
  tagline_en          TEXT,
  tagline_km          TEXT,
  highlight_label_en  TEXT,                             -- e.g. 'Most Popular'
  highlight_label_km  TEXT,

  -- Pricing
  price_cents         INTEGER,                          -- NULL = custom / contact sales
  currency            "Currency" NOT NULL DEFAULT 'USD',
  billing_interval    TEXT NOT NULL DEFAULT 'MONTHLY',  -- 'MONTHLY' | 'ANNUAL' | 'ONE_TIME'

  -- Lifecycle (intentionally two flags — see tables/plans.md Part 7)
  is_active           BOOLEAN NOT NULL DEFAULT TRUE,    -- accepting subscriptions?
  is_public           BOOLEAN NOT NULL DEFAULT FALSE,   -- visible in merchant signup UI?

  -- Presentation
  display_order       INTEGER NOT NULL DEFAULT 0,

  created_at          TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at          TIMESTAMP(3) NOT NULL
);

CREATE INDEX idx_plans_public
  ON plans (display_order)
  WHERE is_public = TRUE AND is_active = TRUE;
```

### `plan_features`

Sibling of `plans`. One row per (plan × capability). `feature_key` is free
TEXT; vocabulary + labels + expected value types live in TypeScript code
(`xfos/contracts/enums/plan-features.ts`), not in the database. `value` is
JSONB — native JSON type preserved (number / boolean / string). Adding a
new feature is an INSERT, not a migration. See `tables/plan-features.md`.

```sql
CREATE TABLE plan_features (
  id            TEXT PRIMARY KEY,
  plan_id       TEXT NOT NULL REFERENCES plans(id) ON DELETE CASCADE,
  feature_key   TEXT NOT NULL,                          -- vocabulary in TS, not DB
  value         JSONB NOT NULL,                         -- number | boolean | string (native JSON type)
  display_order INTEGER NOT NULL DEFAULT 0,
  created_at    TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,

  UNIQUE (plan_id, feature_key)
);

CREATE INDEX ON plan_features (plan_id);
```

### `subscriptions`

Full history table — every subscription period is its own row; at most one
`ACTIVE` row per tenant (enforced by partial unique index). Pricing terms
(`price_cents`, `currency`, `billing_interval`, `plan_code`) are snapshotted
at subscription creation for grandfathering; display names are live-joined
via `plan_id`. Stripe-style billing cycles: `current_period_start` /
`current_period_end` reset on renewal; `started_at` is the lifetime anchor.
Cancellation split: `cancelled_at` = intent timestamp, `cancel_at` =
scheduled end. See `tables/subscriptions.md`.

Stubbed for MVP; fully wired in v1.1+.

```sql
CREATE TABLE subscriptions (
  id                    TEXT PRIMARY KEY,
  tenant_id             TEXT NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,

  -- Plan reference (for display & analytics)
  plan_id               TEXT NOT NULL REFERENCES plans(id),

  -- Snapshot at subscription creation (grandfathering)
  plan_code             TEXT NOT NULL,
  price_cents           INTEGER NOT NULL,                       -- frozen; not re-derived from plans
  currency              "Currency" NOT NULL DEFAULT 'USD',
  billing_interval      TEXT NOT NULL DEFAULT 'MONTHLY',

  -- Lifecycle
  status                "SubscriptionStatus" NOT NULL DEFAULT 'PENDING',

  -- Timeline (Stripe-style)
  started_at            TIMESTAMP(3),                           -- set once, lifetime anchor
  current_period_start  TIMESTAMP(3),                           -- resets each renewal
  current_period_end    TIMESTAMP(3),                           -- resets each renewal

  -- Cancellation (split: intent vs scheduled end)
  cancel_at             TIMESTAMP(3),                           -- scheduled end date (access stops here)
  cancelled_at          TIMESTAMP(3),                           -- when the tenant clicked cancel

  created_at            TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at            TIMESTAMP(3) NOT NULL
);

CREATE INDEX ON subscriptions (tenant_id);

-- Enforce: at most one ACTIVE subscription per tenant
CREATE UNIQUE INDEX subscriptions_one_active_per_tenant
  ON subscriptions (tenant_id)
  WHERE status = 'ACTIVE';

-- Dunning query: PAST_DUE ordered by how close to suspension
CREATE INDEX subscriptions_past_due_idx
  ON subscriptions (current_period_end)
  WHERE status = 'PAST_DUE';
```

### `tenant_sequences`

Per-tenant counters for order and bill numbers. **Daily-reset counter for
orders** (counter resets at the tenant's local midnight, from
`tenant_settings.timezone`). **Running counter for bills** (never resets —
financial/audit). Kitchen tickets inherit the order's number — no separate
counter. Application code MUST NOT write these fields directly; use the
helper functions installed by the hardening migration:

```sql
SELECT allocate_order_number($1);  -- returns 'LB-042' — daily-reset, tenant-prefixed
SELECT allocate_bill_number($1);   -- returns 'LB-B-000125' — running sequential
-- There is NO allocate_ticket_number(). kitchen_tickets.ticket_number is set
-- equal to orders.order_number at ticket creation.
```

See `design-discussions/order-numbering-strategy.md` for the full design.

**2026-04-26 enterprise upgrade:** 3 sanity CHECK constraints
(`next_order_counter >= 1`, `next_bill_number >= 1`,
`counters_reset_on >= '2025-01-01'`) prevent corrupt state from
accidental manual `UPDATE` mistakes or backfill bugs. The
`allocate_order_number()` function body refactored for clarity (same
locking, same atomicity, two simpler statements instead of one
subtle one).

```sql
CREATE TABLE tenant_sequences (
  tenant_id              TEXT PRIMARY KEY REFERENCES tenants(id) ON DELETE CASCADE,

  -- Daily-reset counter for orders (tenant-local midnight)
  next_order_counter     INTEGER NOT NULL DEFAULT 1,
  counters_reset_on      DATE    NOT NULL DEFAULT CURRENT_DATE,  -- tenant-local date

  -- Running counter for bills (never resets — financial/audit)
  next_bill_number       BIGINT  NOT NULL DEFAULT 1,

  updated_at             TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,

  CONSTRAINT tenant_sequences_order_counter_positive
    CHECK (next_order_counter >= 1),
  CONSTRAINT tenant_sequences_bill_number_positive
    CHECK (next_bill_number >= 1),
  CONSTRAINT tenant_sequences_reset_date_sane
    CHECK (counters_reset_on >= DATE '2025-01-01')
);
```

## AUTH — identity, sessions, roles

### `users`

Global identity record; tenant membership is via `user_roles`.
Auth method is via `user_auth_providers` — Telegram, Facebook, or phone-OTP.
Kitchen/counter staff log in via PIN on tenant-bound tablets (future
`user_pins` table, not here).

Design notes (see `design-discussions/authentication-strategy-v2.md`):
- `email` is **nullable** — contact info only, captured from Facebook OAuth
  when available. Not used for login.
- `phone` is **nullable** — E.164 format, present when PHONE provider linked
  (via SMS OTP).
- `avatar_url` is populated from the social provider's profile picture.
- `phone_verified` + `phone_verified_at` — TRUE once the user completes
  SMS OTP for this phone.
- Owner info (personal contact) lives HERE, not on `tenants`.
  Business contact lives on `tenant_settings`.

```sql
CREATE TABLE users (
  id                  TEXT PRIMARY KEY,
  email               TEXT UNIQUE,                        -- nullable: contact info only (captured from Facebook if scope granted); not used for login
  phone               TEXT UNIQUE                         -- nullable: E.164; present when PHONE provider linked
                      CHECK (phone IS NULL OR phone ~ '^\+[1-9][0-9]{6,14}$'),
  full_name           TEXT,
  avatar_url          TEXT,                               -- from social provider profile
  phone_verified      BOOLEAN NOT NULL DEFAULT FALSE,     -- TRUE once SMS OTP completed for this phone
  phone_verified_at   TIMESTAMP(3),                       -- when phone was verified via SMS OTP
  status              "UserStatus" NOT NULL DEFAULT 'PENDING',
  last_login_at       TIMESTAMP(3),
  created_at          TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at          TIMESTAMP(3) NOT NULL
);
```

### `user_auth_providers`

Links a user to one or more external auth providers. One user can have
multiple providers (e.g. Facebook + Telegram + email). Required for the
multi-provider auth strategy — Facebook + Telegram as primary,
email+password as fallback.

Users should link at least 2 providers during onboarding to protect
against single-provider lockout (e.g. Facebook account gets locked).

```sql
CREATE TABLE user_auth_providers (
  id            TEXT PRIMARY KEY,
  user_id       TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  provider      "AuthProvider" NOT NULL,
  provider_id   TEXT NOT NULL,                         -- Facebook user ID, Telegram user ID, or email
  display_name  TEXT,                                  -- name from provider's profile
  avatar_url    TEXT,                                  -- profile pic URL from provider
  metadata      JSONB,                                 -- raw provider response for audit
  linked_at     TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE (provider, provider_id)                       -- one XFOS account per provider identity
);

CREATE INDEX ON user_auth_providers (user_id);
```

### `phone_otp_attempts`

Short-lived audit + rate-limiting store for SMS OTP codes (phone signup,
recovery login, phone update). Keyed by phone number (not user_id) because
OTP flows happen before auth. Never stores OTPs in plaintext — hash only.
Rate limits: 1 send per phone per 60s, 5 per hour; 5 failed attempts per
code → 15-minute phone lockout. See `tables/phone-otp-attempts.md`.

```sql
CREATE TABLE phone_otp_attempts (
  id               TEXT PRIMARY KEY,
  phone            TEXT NOT NULL
                   CHECK (phone ~ '^\+[1-9][0-9]{6,14}$'),
  otp_hash         TEXT NOT NULL,                          -- SHA-256 of OTP + per-row salt
  purpose          TEXT NOT NULL,                          -- 'signup' | 'login' | 'reverify' | 'add_phone'
  expires_at       TIMESTAMP(3) NOT NULL,                  -- NOW() + 5 min
  attempt_count    INTEGER NOT NULL DEFAULT 0,             -- failed verifications
  used_at          TIMESTAMP(3),                           -- set when successfully consumed (single-use)
  ip_address       INET,
  user_agent       TEXT,
  created_at       TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- Hot path: find the active OTP for a phone during verification
CREATE INDEX phone_otp_attempts_phone_active_idx
  ON phone_otp_attempts (phone, created_at DESC)
  WHERE used_at IS NULL AND expires_at > CURRENT_TIMESTAMP;

-- Rate-limit queries
CREATE INDEX phone_otp_attempts_phone_created_idx
  ON phone_otp_attempts (phone, created_at DESC);

-- Cleanup job
CREATE INDEX phone_otp_attempts_expires_idx
  ON phone_otp_attempts (expires_at)
  WHERE used_at IS NULL;
```

### `user_roles`

Junction: user ↔ tenant ↔ role. `PLATFORM_ADMIN` and `PLATFORM_STAFF`
have `tenant_id = NULL`. The unique constraint uses `NULLS NOT DISTINCT`
(PG 15+) so that a user cannot be assigned a platform role twice — two
`(user, NULL, PLATFORM_ADMIN)` rows would otherwise both satisfy the
default `UNIQUE` semantics.

```sql
CREATE TABLE user_roles (
  id         TEXT PRIMARY KEY,
  user_id    TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  tenant_id  TEXT REFERENCES tenants(id) ON DELETE CASCADE,
  role       "Role" NOT NULL,
  created_at TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE NULLS NOT DISTINCT (user_id, tenant_id, role)
);

CREATE INDEX ON user_roles (user_id);
CREATE INDEX ON user_roles (tenant_id);
```

### `refresh_tokens`

Store the SHA-256 hash of the raw token, never plaintext.

```sql
CREATE TABLE refresh_tokens (
  id         TEXT PRIMARY KEY,
  user_id    TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  tenant_id  TEXT REFERENCES tenants(id) ON DELETE CASCADE,
```

### `qr_contexts`

Scanned by customers to resolve tenant + (optional) table. `table_id`
(composite FK to `tables`) replaced the old `table_ref TEXT` on
2026-04-24.

**2026-04-25 enterprise upgrade:** added `version` (OCC for concurrent
relabel/deactivate/regen); deactivation accountability triad
(`deactivated_at` + `deactivated_by_id` + `deactivation_reason`) using
new sibling enum `QrDeactivationReason`; `created_by_id`; regeneration
chain (`replaces_id` self-FK); analytics counters (`scan_count` +
`last_scanned_at`, `print_count` + `last_printed_at`); merchant `notes`;
6 new CHECK constraints (lifecycle gating, sanity, count nonneg).

```sql
CREATE TABLE qr_contexts (
  tenant_id    TEXT NOT NULL,
  id           TEXT NOT NULL,

  -- Provenance
  table_id     TEXT,                                          -- composite FK; required when context_type='TABLE'
  replaces_id  TEXT,                                          -- composite self-FK; points to QR this row supersedes (regen chain)

  -- Public identity
  token        TEXT UNIQUE NOT NULL,                          -- globally unique URL slug
  label        TEXT,                                          -- merchant-facing label
  notes        TEXT,                                          -- merchant-facing free-form notes
  context_type "QrContextType" NOT NULL DEFAULT 'STOREFRONT',

  -- Lifecycle
  is_active    BOOLEAN NOT NULL DEFAULT TRUE,
  expires_at   TIMESTAMP(3),

  -- Optimistic concurrency
  version      INTEGER NOT NULL DEFAULT 1,

  -- Lifecycle accountability
  created_by_id        TEXT REFERENCES users(id),             -- single-column FK: users is global; NULL when auto-provisioned at onboarding
  deactivated_at       TIMESTAMP(3),
  deactivated_by_id    TEXT REFERENCES users(id),             -- NULL when reason is EXPIRED_AUTO or TENANT_DEACTIVATED (system actor)
  deactivation_reason  "QrDeactivationReason",

  -- Analytics counters
  scan_count           INTEGER NOT NULL DEFAULT 0,
  last_scanned_at      TIMESTAMP(3),
  print_count          INTEGER NOT NULL DEFAULT 0,
  last_printed_at      TIMESTAMP(3),

  -- Timestamps
  created_at   TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at   TIMESTAMP(3) NOT NULL,

  PRIMARY KEY (tenant_id, id),
  FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON DELETE CASCADE,
  FOREIGN KEY (tenant_id, table_id)    REFERENCES tables(tenant_id, id),
  FOREIGN KEY (tenant_id, replaces_id) REFERENCES qr_contexts(tenant_id, id),

  -- Context type vs table_id pairing
  CONSTRAINT qr_contexts_table_kind_requires_table_id
    CHECK ((context_type != 'TABLE') OR (table_id IS NOT NULL)),
  CONSTRAINT qr_contexts_storefront_kind_has_no_table_id
    CHECK ((context_type != 'STOREFRONT') OR (table_id IS NULL)),

  -- Sanity
  CONSTRAINT qr_contexts_expires_after_created
    CHECK (expires_at IS NULL OR expires_at > created_at),
  CONSTRAINT qr_contexts_replaces_not_self
    CHECK (replaces_id IS NULL OR replaces_id != id),
  CONSTRAINT qr_contexts_scan_count_nonneg     CHECK (scan_count  >= 0),
  CONSTRAINT qr_contexts_print_count_nonneg    CHECK (print_count >= 0),

  -- Lifecycle/accountability gating
  CONSTRAINT qr_contexts_active_no_deactivation
    CHECK ((is_active = FALSE)
           OR (deactivated_at IS NULL AND deactivated_by_id IS NULL AND deactivation_reason IS NULL)),
  CONSTRAINT qr_contexts_inactive_has_reason
    CHECK ((is_active = TRUE)
           OR (deactivated_at IS NOT NULL AND deactivation_reason IS NOT NULL)),
  CONSTRAINT qr_contexts_human_reasons_have_actor
    CHECK ((deactivation_reason IS NULL)
           OR (deactivation_reason IN ('EXPIRED_AUTO', 'TENANT_DEACTIVATED'))
           OR (deactivated_by_id IS NOT NULL))
);

CREATE INDEX ON qr_contexts (tenant_id) WHERE is_active = TRUE;

-- At most ONE active QR per table (regeneration deactivates the old one
-- before creating the new one, in a single transaction).
CREATE UNIQUE INDEX qr_contexts_one_active_per_table
  ON qr_contexts (tenant_id, table_id)
  WHERE is_active = TRUE AND table_id IS NOT NULL;

-- Stale-QR cleanup nudge ("hasn't been scanned in 6 months")
CREATE INDEX ON qr_contexts (tenant_id, last_scanned_at)
  WHERE is_active = TRUE;

-- Regen chain walk
CREATE INDEX ON qr_contexts (tenant_id, replaces_id)
  WHERE replaces_id IS NOT NULL;
```

### `order_sessions`

Dine-in / open-tab session. Closed by payment or merchant action — no
timer-based expiry. A background cleanup job closes abandoned sessions
(no activity for 24h) platform-wide.

**2026-04-25 enterprise upgrade:** added `version` (OCC for multi-writer
races); `closed_reason` ("OrderSessionCloseReason" sibling enum) +
matching CHECK constraints; `last_activity_at` (drives the cleanup job
correctly — increments on every order/cart write, so steady-ordering
sessions aren't garbage-collected); per-transition accountability
(`opened_by_id`, `server_id`, `closed_by_id`); operational metadata
(`party_size`, `notes`); denormalized session running totals
(`subtotal_cents`, `total_cents`, `order_count`) so the merchant-portal
"occupied tables" view is a single-row read. Partial unique index
enforces "one ACTIVE session per table."

```sql
CREATE TABLE order_sessions (
  tenant_id           TEXT NOT NULL,
  id                  TEXT NOT NULL,

  -- Provenance
  qr_context_id       TEXT,                                   -- nullable; composite FK below
  table_id            TEXT,                                   -- nullable; live composite FK to tables
  table_ref           TEXT,                                   -- snapshot of tables.label at session-open time

  -- Lifecycle
  status              "OrderSessionStatus" NOT NULL DEFAULT 'ACTIVE',
  closed_reason       "OrderSessionCloseReason",              -- nullable; required when status=CLOSED

  -- Optimistic concurrency
  version             INTEGER NOT NULL DEFAULT 1,

  -- Operational metadata
  party_size          SMALLINT,                                -- nullable; party size set by host
  notes               TEXT,

  -- Running session totals (denormalized for floor-plan view)
  subtotal_cents      INTEGER NOT NULL DEFAULT 0,
  total_cents         INTEGER NOT NULL DEFAULT 0,
  order_count         INTEGER NOT NULL DEFAULT 0,

  -- Lifecycle timestamps
  opened_at           TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  last_activity_at    TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  closed_at           TIMESTAMP(3),

  -- Accountability (single-column FKs: users is global)
  opened_by_id        TEXT REFERENCES users(id),               -- staff who manually opened (NULL when QR scan)
  server_id           TEXT REFERENCES users(id),               -- staff serving this table
  closed_by_id        TEXT REFERENCES users(id),               -- staff who closed

  PRIMARY KEY (tenant_id, id),
  FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON DELETE CASCADE,
  FOREIGN KEY (tenant_id, qr_context_id) REFERENCES qr_contexts(tenant_id, id),
  FOREIGN KEY (tenant_id, table_id)      REFERENCES tables(tenant_id, id),

  CONSTRAINT order_sessions_closed_at_matches_status
    CHECK ((closed_at IS NULL) = (status = 'ACTIVE')),
  CONSTRAINT order_sessions_closed_at_after_opened
    CHECK (closed_at IS NULL OR closed_at >= opened_at),
  CONSTRAINT order_sessions_closed_reason_only_when_closed
    CHECK ((status = 'CLOSED') = (closed_reason IS NOT NULL)),
  CONSTRAINT order_sessions_party_size_positive
    CHECK (party_size IS NULL OR party_size > 0),
  CONSTRAINT order_sessions_subtotal_nonneg     CHECK (subtotal_cents >= 0),
  CONSTRAINT order_sessions_total_nonneg        CHECK (total_cents    >= 0),
  CONSTRAINT order_sessions_order_count_nonneg  CHECK (order_count    >= 0),
  CONSTRAINT order_sessions_last_activity_after_opened
    CHECK (last_activity_at >= opened_at)
);

CREATE INDEX ON order_sessions (tenant_id, status);

-- One ACTIVE session per table
CREATE UNIQUE INDEX order_sessions_one_active_per_table
  ON order_sessions (tenant_id, table_id)
  WHERE status = 'ACTIVE' AND table_id IS NOT NULL;

-- Cleanup job: find stale ACTIVE sessions
CREATE INDEX order_sessions_cleanup_idx
  ON order_sessions (tenant_id, last_activity_at)
  WHERE status = 'ACTIVE';

-- Historical session list (merchant portal "all sessions" view)
CREATE INDEX ON order_sessions (tenant_id, opened_at DESC);
```

### `idempotency_keys`

24h TTL. Use `INSERT … ON CONFLICT` to short-circuit replayed requests.
Uniqueness is scoped per tenant (C5) to prevent cross-tenant cache-hit exfil.
Cleanup: `cleanup_expired_idempotency_keys()` runs hourly via BullMQ (C4).

**2026-04-26 enterprise upgrade:** added `request_body_hash` (SHA-256
hex of the request body) so retries with the same key but a *different*
body are detected and rejected with 409 — closes a real security hole;
applied actor-triad mirror (`actor_type`/`actor_label`/`user_id`) +
matching CHECK constraints; added `request_id` for cross-event
correlation; added 4 CHECK constraints (USER⇔user_id,
non-USER⇒actor_label, `expires_at > created_at`, valid HTTP code);
added 2 partial indexes (request_id, user_id).

```sql
CREATE TABLE idempotency_keys (
  tenant_id          TEXT NOT NULL,
  id                 TEXT NOT NULL,

  -- Request identity
  key                TEXT NOT NULL,                            -- client-generated (typically UUID v4)
  endpoint           TEXT NOT NULL,                            -- e.g. 'POST /storefront/orders'
  request_body_hash  TEXT NOT NULL,                            -- SHA-256 hex of request body

  -- Cached response
  response_code      INTEGER NOT NULL,
  response_body      JSONB NOT NULL,

  -- Actor (aligned with audit_logs / order_status_history / kitchen_ticket_events)
  actor_type         "AuditActorType" NOT NULL,
  actor_label        TEXT,                                     -- required when actor_type != USER
  user_id            TEXT REFERENCES users(id),                -- single-column FK; users is global

  -- Correlation
  request_id         TEXT,                                     -- API request ID

  -- Lifecycle
  created_at         TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  expires_at         TIMESTAMP(3) NOT NULL,

  PRIMARY KEY (tenant_id, id),
  FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON DELETE CASCADE,
  UNIQUE (tenant_id, key, endpoint),

  CONSTRAINT idempotency_keys_user_actor_has_user_id
    CHECK ((actor_type = 'USER') = (user_id IS NOT NULL)),
  CONSTRAINT idempotency_keys_system_actors_have_label
    CHECK ((actor_type = 'USER') OR (actor_label IS NOT NULL)),
  CONSTRAINT idempotency_keys_expires_after_created
    CHECK (expires_at > created_at),
  CONSTRAINT idempotency_keys_response_code_valid
    CHECK (response_code BETWEEN 100 AND 599)
);

  WHERE request_id IS NOT NULL;
```

enum) + `actor_label` with matching CHECK constraints; added
`cancellation_reason "OrderCancellationReason"` typed mirror for
CANCELLED transitions; added `request_id` for cross-event correlation;
added CHECK forbidding no-op STATUS_CHANGE transitions; added 2 partial
indexes (request_id, CANCELLED-segment).

```sql
CREATE TABLE kitchen_ticket_events (
  tenant_id            TEXT NOT NULL,
  id                   TEXT NOT NULL,
  ticket_id            TEXT NOT NULL,

  -- Event taxonomy
  event_type           TEXT NOT NULL DEFAULT 'STATUS_CHANGE',  -- 'STATUS_CHANGE' | 'EXPEDITE' | 'PRINT' | 'PRIORITY_CHANGE'
  from_status          TEXT,
  to_status            TEXT,                                   -- nullable when event_type != STATUS_CHANGE
  cancellation_reason  "OrderCancellationReason",              -- typed mirror when to_status = CANCELLED

  -- Actor (aligned with audit_logs and order_status_history)
  actor_type           "AuditActorType" NOT NULL,
  actor_label          TEXT,                                   -- required when actor_type != USER
  changed_by_id        TEXT REFERENCES users(id),              -- single-column FK; users is global

  -- Correlation
  request_id           TEXT,                                   -- API request / BullMQ job ID
  reason               TEXT,                                   -- free-form human-readable note
  metadata             JSONB,                                  -- event-type-specific structured payload

  created_at           TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,

  PRIMARY KEY (tenant_id, id),
  FOREIGN KEY (tenant_id) REFERENCES tenants(id) ON DELETE CASCADE,
  FOREIGN KEY (tenant_id, ticket_id) REFERENCES kitchen_tickets(tenant_id, id) ON DELETE CASCADE,

  CONSTRAINT kitchen_ticket_events_status_change_has_to_status
    CHECK ((event_type != 'STATUS_CHANGE') OR (to_status IS NOT NULL)),
  CONSTRAINT kitchen_ticket_events_user_actor_has_user_id
    CHECK ((actor_type = 'USER') = (changed_by_id IS NOT NULL)),
  CONSTRAINT kitchen_ticket_events_system_actors_have_label
    CHECK ((actor_type = 'USER') OR (actor_label IS NOT NULL)),
  CONSTRAINT kitchen_ticket_events_no_op_status_change
    CHECK ((event_type != 'STATUS_CHANGE')
           OR (from_status IS NULL OR from_status != to_status))
);

CREATE INDEX ON kitchen_ticket_events (tenant_id, ticket_id, created_at);

-- Correlation lookup
CREATE INDEX kitchen_ticket_events_request_id_idx
  ON kitchen_ticket_events (request_id)
  WHERE request_id IS NOT NULL;

-- Cancellation analytics
CREATE INDEX kitchen_ticket_events_cancellation_idx
  ON kitchen_ticket_events (tenant_id, created_at DESC)
  WHERE to_status = 'CANCELLED';
```

## ADMIN / PLATFORM — audit logs

### `audit_logs`

Append-only, cross-tenant. `action` is a free-form string (e.g.
`order.created`, `bill.paid`, `user.role_changed`); vocabulary is documented
in the docs, not as a Prisma enum. `tenant_id` is NULL for platform-level events.

**2026-04-26 enterprise upgrade:** added typed `category`/`severity`/
`actor_type` enums (action stays free-form for richness, but coarse axes
are now indexable); `actor_label` (system-actor name when not USER);
correlation IDs (`request_id`, `auth_session_id`, `idempotency_key`);
first-class state diff (`previous_state`, `new_state`) split out of
`metadata`; `user_agent`; `retention_until`; 3 CHECK constraints
(actor-typing) and 5 new indexes (action, user, severity-alert,
request_id, category).

```sql
CREATE TABLE audit_logs (
  id              TEXT PRIMARY KEY,

  -- Tenant scope (nullable for platform-level events)
  tenant_id       TEXT REFERENCES tenants(id) ON DELETE SET NULL,

  -- Categorization
  category        "AuditCategory" NOT NULL,                  -- coarse domain axis
  severity        "AuditSeverity" NOT NULL DEFAULT 'INFO',
  action          TEXT NOT NULL,                             -- free-form 'entity.verb'

  -- Actor
  actor_type      "AuditActorType" NOT NULL,
  actor_label     TEXT,                                      -- required when actor_type != USER
  user_id         TEXT REFERENCES users(id) ON DELETE SET NULL,

  -- Subject
  entity_type     TEXT,                                      -- e.g. 'Order', 'Bill', 'Tenant'
  entity_id       TEXT,                                      -- cuid of subject

  -- State diff (first-class for the common shape)
  previous_state  JSONB,
  new_state       JSONB,
  metadata        JSONB,                                     -- everything else

  -- Correlation
  request_id      TEXT,                                      -- API request ID / BullMQ job ID
  auth_session_id TEXT,                                      -- JWT session that authorized USER actions
  idempotency_key TEXT,                                      -- request idempotency-key when applicable

  -- Provenance
  ip_address      INET,
  user_agent      TEXT,

  -- Lifecycle
  retention_until TIMESTAMP(3),                              -- NULL = retain forever
  created_at      TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,

  CONSTRAINT audit_logs_user_actor_has_user_id
    CHECK ((actor_type = 'USER') = (user_id IS NOT NULL)),
  CONSTRAINT audit_logs_system_actors_have_label
    CHECK ((actor_type = 'USER') OR (actor_label IS NOT NULL)),
  CONSTRAINT audit_logs_entity_id_requires_type
    CHECK (entity_id IS NULL OR entity_type IS NOT NULL)
);

-- Tenant timeline (existing)
CREATE INDEX ON audit_logs (tenant_id, created_at DESC);
-- Entity history (existing)
CREATE INDEX ON audit_logs (entity_type, entity_id);
-- Action-pattern queries: "all bill.voided this week"
CREATE INDEX ON audit_logs (action, created_at DESC);
-- Category-scoped dashboards
CREATE INDEX ON audit_logs (category, created_at DESC);
-- "What did this user do?" — investigation pattern
CREATE INDEX audit_logs_user_id_created_at_idx
  ON audit_logs (user_id, created_at DESC)
  WHERE user_id IS NOT NULL;
-- Alert feed
CREATE INDEX audit_logs_severity_alert_idx
  ON audit_logs (severity, created_at DESC)
  WHERE severity IN ('WARNING', 'ALERT');
-- Correlation lookup
CREATE INDEX audit_logs_request_id_idx
  ON audit_logs (request_id)
  WHERE request_id IS NOT NULL;
-- Retention cleanup job
CREATE INDEX audit_logs_retention_idx
  ON audit_logs (retention_until)
  WHERE retention_until IS NOT NULL;
```

> **Partitioning is explicitly deferred.** `audit_logs` grows unboundedly
> in principle, but Postgres handles 100M+ rows in a single table fine.
> Partitioning by `created_at` (monthly RANGE partitions) is the right
> answer when row counts force it, but the operational complexity isn't
> justified at MVP. Document this decision; revisit when a real write-rate
> projection exists.

## Table inventory (38 tables)

> **Composite-PK convention (since 2026-04-24, fully applied 2026-04-25):** every tenant-scoped table uses `PRIMARY KEY (tenant_id, id)` and every cross-tenant-scoped FK is composite `FOREIGN KEY (tenant_id, x_id) REFERENCES x(tenant_id, id)`. This makes cross-tenant linking impossible by construction — **no parity triggers anywhere in the schema**. Non-tenant-scoped tables (`tenants`, `users`, `user_auth_providers`, `refresh_tokens`, `phone_otp_attempts`, `plans`, `plan_features`, `subscriptions`) keep single-column `id` PKs. Junction tables with nullable tenant_id (`user_roles`, `audit_logs`) also keep single-column `id` PKs and rely on application-layer scoping.
