# `tenants`

| Attribute | Value |
|---|---|

| **Tenant-scoped?** | No -- this table _is_ the tenant |

---

## Part 1: Overview

The `tenants` table is the root entity of the entire XFOS platform. Every business that signs up -- whether it is a noodle stall in Phsar Thmei, a boba kiosk at AEON Mall, or a dine-in restaurant on Pub Street in Siem Reap -- gets exactly one row here. This table holds **identity only**: a machine-generated ID, a human-readable slug for URLs, a bilingual display name, and a lifecycle status. Everything else -- operational settings, contact info, address, branding, payment rails -- lives in `tenant_settings`. Owner details live on `users` + `user_roles`, not here. This separation means the tenant record outlives any individual owner: owners come and go, but the business persists.

## Part 2: CREATE TABLE

```sql
CREATE TABLE tenants (
  id          TEXT PRIMARY KEY,
  slug        TEXT UNIQUE NOT NULL,                           -- URL key (kebab-case, long form)
  code_prefix TEXT UNIQUE NOT NULL                            -- SHORT prefix for order/bill numbers ('LB', 'BQ')
              CHECK (code_prefix ~ '^[A-Z]{2,4}$'),           -- 2-4 uppercase Latin letters, immutable
  name_en     TEXT NOT NULL,                                  -- English display name (primary, system default)
  name_km     TEXT,                                           -- Khmer display name (optional)
  status      "TenantStatus" NOT NULL DEFAULT 'DRAFT',
  created_at  TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at  TIMESTAMP(3) NOT NULL
);
```

Referenced enum:

```sql
CREATE TYPE "TenantStatus" AS ENUM ('DRAFT', 'ACTIVE', 'SUSPENDED', 'ARCHIVED');
```

## Part 3: Column-by-Column

### `id` -- TEXT PRIMARY KEY

- **Type:** `TEXT`
- **Nullable:** No
- **Default:** `cuid()` (generated by the application, not the database)
- **Purpose:** Globally unique internal identifier for the tenant. Used as the foreign key target in every tenant-scoped table across the schema (22+ tables reference `tenants(id)`).
- **Constraints:** Primary key.
- **Why it exists:** The platform needs a stable, non-guessable, system-internal identifier. CUIDs are collision-resistant, sortable by creation time, and safe for distributed generation (no central sequence server). This ID is never shown in URLs or to end users -- the `slug` serves that role.

### `slug` -- TEXT UNIQUE NOT NULL

- **Type:** `TEXT`
- **Nullable:** No
- **Default:** None (set by the application during tenant creation)
- **Purpose:** Human-readable, URL-safe identifier. Appears in storefront URLs (e.g., `xfos.com/t/lucky-burger-pp`).
- **Constraints:** `UNIQUE` (platform-wide).
- **Why it exists:** Customers see this in the URL when scanning a QR code and landing on the storefront. Merchants see it in their portal URL. It must be unique, readable, and stable (changing it would break bookmarks and QR codes). Examples: `lok-lak-house`, `boba-queen-sr`, `street-99-noodles`.

### `code_prefix` -- TEXT UNIQUE NOT NULL

- **Type:** `TEXT`
- **Nullable:** No
- **Default:** None (chosen by the tenant at onboarding)
- **Purpose:** Short (2-4 character) uppercase identifier used as the prefix on every customer-facing order / ticket number (`LB-042`) and on bill numbers (`LB-B-000125`). Different from `slug`: `slug` is long-form kebab-case for URLs; `code_prefix` is short, all-caps, pronounceable for verbal exchange ("I'm order LB-042").
- **Constraints:** `UNIQUE` (platform-wide), `NOT NULL`, `CHECK (code_prefix ~ '^[A-Z]{2,4}$')` — uppercase Latin letters only, no digits, no punctuation. **Immutable**: once set, never changed, even under a tenant rename or rebrand.
- **Why it exists:** Multi-tenant order numbers must be unambiguous across tenants. Without the prefix, `042` at Lucky Burger and `042` at Boba Queen are indistinguishable in cross-tenant contexts (platform admin, cross-store support, audit logs). With it, `LB-042` and `BQ-042` are globally unique just from their format. Examples: `LB` (Lucky Burger), `BQ` (Boba Queen), `PPN` (Phnom Penh Noodles), `SKO` (Sach Ko Angkor).
- **Why all-uppercase Latin:** readable on low-resolution kitchen tablets and receipts; no confusion between `0/O`, `1/l/I`, or punctuation lookalikes; no Unicode edge cases.
- **Why 2-4 chars:** 2 gives ~676 combinations (plenty for MVP), 4 gives ~450K. Short enough to keep the full order number under 7 characters (`LBCD-999`).
- **Chosen at onboarding:** the tenant picks their prefix with a real-time availability check (UX similar to picking a username). If the desired prefix is taken, the onboarding UI suggests alternatives. Once confirmed and the tenant is created, it is frozen.
- **Why immutable:** historical references (`LB-042` in last month's audit log, in a customer support ticket, on a printed receipt) must stay resolvable. A rebrand of the business does not renumber or rename its history. See `design-discussions/order-numbering-strategy.md` for the full rationale.

### `name_en` -- TEXT NOT NULL

- **Type:** `TEXT`
- **Nullable:** No
- **Default:** None
- **Purpose:** The business's English display name. English is the primary system language -- used in admin interfaces, internal tooling, platform-admin dashboards, billing communications, and as the fallback when Khmer is not available.
- **Constraints:** `NOT NULL`.
- **Why it exists:** Every tenant must have at least an English name. This is the name that appears in platform admin lists, subscription invoices, and system logs. Two separate columns (`name_en` + `name_km`) were chosen over a JSON blob or translation table because each tenant has exactly one name in at most two locales -- the simplicity of two columns outweighs the flexibility of a translation mechanism here.

### `name_km` -- TEXT (nullable)

- **Type:** `TEXT`
- **Nullable:** Yes
- **Default:** `NULL`
- **Purpose:** The business's Khmer display name. Shown on the customer-facing storefront (which defaults to Khmer locale), on receipts, and in the public "About" section.
- **Constraints:** None beyond type.
- **Why it exists:** Cambodia is a Khmer-first market. A restaurant named "Lucky Burger" in English might present itself as "ឡាកគី បឺហ្គឺ" to local customers. However, some businesses (especially those targeting tourists or expats) may operate without a Khmer name, so it is nullable.

### `status` -- "TenantStatus" NOT NULL DEFAULT 'DRAFT'

- **Type:** `TenantStatus` enum
- **Nullable:** No
- **Default:** `'DRAFT'`
- **Purpose:** Lifecycle state of the tenant. Controls whether the storefront is accessible, whether orders can be placed, and whether the tenant appears in platform admin lists as active.
- **Constraints:** Must be one of: `DRAFT`, `ACTIVE`, `SUSPENDED`, `ARCHIVED`.
- **Why it exists:** A new tenant starts in `DRAFT` while the owner completes onboarding (filling out `tenant_settings`, creating menu items, configuring payments). Once `setup_progress.go_live_ready` is true and the owner clicks "Go Live", the status transitions to `ACTIVE`. `SUSPENDED` is for non-payment or policy violations (storefront returns a "temporarily unavailable" page). `ARCHIVED` is the soft-delete end state -- the tenant row stays for audit purposes, but all operational access is blocked.

### `created_at` -- TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP

- **Type:** `TIMESTAMP(3)` (millisecond precision)
- **Nullable:** No
- **Default:** `CURRENT_TIMESTAMP`
- **Purpose:** When the tenant row was first created. Used in platform admin reports ("tenants onboarded this month"), audit trails, and sorting.
- **Constraints:** `NOT NULL`, auto-populated.
- **Why it exists:** Standard audit timestamp. Every table in the schema carries `created_at`.

### `updated_at` -- TIMESTAMP(3) NOT NULL

- **Type:** `TIMESTAMP(3)`
- **Nullable:** No
- **Default:** Managed by Prisma's `@updatedAt` directive (set automatically on every write)
- **Purpose:** When the tenant record was last modified. Useful for detecting stale cache entries and debugging.
- **Constraints:** `NOT NULL`.
- **Why it exists:** Standard audit timestamp. Prisma handles it automatically; application code never sets it.

## Part 4: Indexes

### Primary key index on `id`

- **Implicit:** Yes (created automatically by `PRIMARY KEY`)
- **Query served:** Any lookup by tenant ID, which is every tenant-scoped query in the system since all child tables join or filter on `tenant_id`.
- **Example:**
  ```sql
  SELECT * FROM tenants WHERE id = 'clx7abc123def';
  ```

### Unique index on `slug`

- **Implicit:** Yes (created automatically by `UNIQUE`)
- **Query served:** Storefront URL resolution. When a customer scans a QR code, the URL contains the slug, and the backend resolves it to a tenant ID.
- **Example:**
  ```sql
  SELECT id, name_en, status FROM tenants WHERE slug = 'lucky-burger-pp';
  ```

## Part 5: Relationships

The `tenants` table is the root of the entire schema graph. It has no outgoing foreign keys -- it is only referenced by others.

### Incoming references (child tables)

| Child table | FK column | On Delete | Why |
|---|---|---|---|
| `tenant_settings` | `tenant_id` | `CASCADE` | Settings are meaningless without the tenant |
| `setup_progress` | `tenant_id` | `CASCADE` | Onboarding state is per-tenant |
| `subscriptions` | `tenant_id` | `CASCADE` | Billing records belong to the tenant |
| `tenant_sequences` | `tenant_id` | `CASCADE` | Counters are per-tenant |
| `user_roles` | `tenant_id` | `CASCADE` | Role assignments are scoped to a tenant |
| `refresh_tokens` | `tenant_id` | `CASCADE` | Tokens are tenant-scoped |
| `invitations` | `tenant_id` | `CASCADE` | Invitations belong to a tenant |
| `menu_categories` | `tenant_id` | `CASCADE` | Catalog is per-tenant |
| `menu_items` | `tenant_id` | `CASCADE` | Menu items are per-tenant |
| `orders` | `tenant_id` | `CASCADE` | Orders belong to a tenant |
| `bills` | `tenant_id` | `CASCADE` | Bills belong to a tenant |
| `payments` | `tenant_id` | `CASCADE` | Payments belong to a tenant |
| `kitchen_tickets` | `tenant_id` | `CASCADE` | Tickets are per-tenant |
| `audit_logs` | `tenant_id` | `SET NULL` | Audit logs survive tenant deletion for platform compliance |

All child FKs use `ON DELETE CASCADE` except `audit_logs`, which uses `ON DELETE SET NULL` to preserve the audit trail even after a tenant is permanently deleted.

## Part 6: Real-World Usage Scenarios

### Scenario 1: A new noodle stall registers on XFOS

Sokha opens "Street 99 Noodles" (ផ្ទះម្ហូបផ្លូវ ៩៩), a noodle stall near Riverside in Phnom Penh. She visits the XFOS registration page and fills in her business name.

The system:
1. Generates a cuid for `id` (e.g., `clx8k9m2n0001vq...`).
2. Derives a slug: `street-99-noodles`.
3. **Prompts Sokha to pick a `code_prefix`** — the short identifier that will appear on every order her customers see (`S99-042`, etc.). The onboarding UI shows a real-time availability check. Sokha picks `S99`; the system confirms it is available and reserves it.
4. Inserts `name_en = 'Street 99 Noodles'`, `name_km = 'ផ្ទះម្ហូបផ្លូវ ៩៩'`, `code_prefix = 'S99'`.
5. Sets `status = 'DRAFT'`.
6. The `tenants_create_sequences` trigger auto-creates a row in `tenant_sequences` with counters at default values.
7. A `tenant_settings` row is created with defaults (`service_model = 'STALL_KIOSK'`, `currency = 'USD'`, `default_locale = 'km'`, `timezone = 'Asia/Phnom_Penh'`).
8. A `setup_progress` row is created with all milestone timestamps `NULL`.
9. A `user_roles` row links Sokha's user account as `TENANT_OWNER`.

The storefront is not yet accessible because `status = 'DRAFT'`. When Sokha's first customer places an order after go-live, its number will be `S99-001`, resetting daily thereafter.

### Scenario 2: Platform admin suspends a tenant for policy violation

"Quick Bite Siem Reap" has been reported for selling items not on their declared menu. A platform admin reviews the report and suspends the tenant.

```sql
UPDATE tenants
SET    status = 'SUSPENDED', updated_at = NOW()
WHERE  slug = 'quick-bite-sr';
```

The storefront now returns a "This restaurant is temporarily unavailable" page. The kitchen app shows a banner. The merchant portal shows a notice with a support contact link. No data is deleted -- the tenant can be reactivated by changing status back to `ACTIVE`.

### Scenario 3: Viewing all active tenants in the platform admin dashboard

A platform operator opens the admin dashboard to see how many businesses are live.

```sql
SELECT id, slug, code_prefix, name_en, name_km, created_at
FROM   tenants
WHERE  status = 'ACTIVE'
ORDER BY created_at DESC;
```

This returns all live businesses, sorted newest first. The admin can click into any one to see their `tenant_settings`, recent orders, and subscription status.

## Part 7: Design Decisions

### Why no owner info on this table

Early designs considered putting `owner_name`, `owner_email`, and `owner_phone` directly on `tenants`. This was rejected because:
- A tenant can have multiple owners (via `user_roles` with `role = 'TENANT_OWNER'`).
- Owners change over time (one leaves, another joins).
- The tenant record should outlive any individual person.
- The owner's personal email is not the same as the business contact email (which lives on `tenant_settings.business_email`).

### Why two name columns instead of a translation table

Each tenant has exactly one name in at most two locales (English and Khmer). A `tenant_name_translations` table would add a join, add complexity, and buy nothing -- tenants will never have names in 15 languages. Two columns (`name_en`, `name_km`) are the simplest correct representation.

### Why both `slug` and `code_prefix` — they are NOT redundant

`slug` (long-form kebab-case, e.g. `street-99-noodles`) and `code_prefix` (short uppercase, e.g. `S99`) serve different audiences:

| | `slug` | `code_prefix` |
|---|---|---|
| **Primary use** | URL routing (`xfos.app/t/street-99-noodles`) | Order / bill number prefix (`S99-042`, `S99-B-000125`) |
| **Optimized for** | Search engines, bookmarks, humans reading URLs | Saying aloud, fitting on a kitchen tablet, receipts |
| **Length** | 10-40 chars | 2-4 chars |
| **Case** | lowercase with dashes | UPPERCASE letters only |
| **Visibility** | URL bar | Every receipt, every kitchen screen, every audit log line |

An earlier iteration proposed making `slug` do both jobs. Rejected because `street-99-noodles-042` is too long to read aloud and too clunky on a receipt; shortening the slug to `s99` would degrade SEO / URL readability. Two dedicated columns, each optimized for one job, is simpler than compromising either.

### Why `slug` serves double duty as the URL key and tenant code

An early proposal added a separate `tenant_key` column (e.g., `T-30303948`) to serve as a typed identifier during login, framed as "2FA." This was rejected because it is not real two-factor authentication and would cause UX friction. The slug already provides a human-readable identifier, and the internal `id` handles all machine-to-machine references.

### Why status uses an enum, not a boolean `is_active`

A boolean cannot represent the full lifecycle: `DRAFT` (not yet launched), `ACTIVE` (live), `SUSPENDED` (temporarily blocked), `ARCHIVED` (permanently retired). The enum captures these distinct states and enables clear guard clauses in the application (e.g., "only allow orders if status = ACTIVE").

### H7 hardening: `deleted_at` column

The high-tier hardening migration (`20260411_mvp_hardening_high_tier.sql`) adds a `deleted_at TIMESTAMPTZ` column for staged tenant deletion. Rather than cascading a `DELETE` across 22+ child tables in one transaction (which would lock everything), the app sets `deleted_at`, then a BullMQ worker calls `permanent_delete_tenant()` to clean up child rows in reverse dependency order. This column is not yet in the base Prisma schema but will be added after the hardening migrations are applied.
