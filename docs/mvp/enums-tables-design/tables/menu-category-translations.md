# Table Reference: `menu_category_translations`

| Property | Value |
|---|---|
| **Domain** | Catalog |
| **Tenant-scoped** | Yes (denormalized -- C1 finding) |
| **Prisma model** | `MenuCategoryTranslation` |
| **Mapped table** | `menu_category_translations` |
| **Soft delete** | No (cascades with parent category) |

---

## Part 1: Overview

`menu_category_translations` stores the localized display names for menu
categories. Each row holds a single translation of a category name in one
locale (`en` or `kh`). The constraint `UNIQUE(category_id, locale)` guarantees
at most one translation per locale per category.

**Why a translation table instead of inline columns:**

The `tenants` table uses inline `name_en` / `name_km` columns because a tenant
has exactly one name in at most two known locales. Categories use a separate
table because:

1. **Queryability.** The merchant portal onboarding wizard needs to ask "which
   categories are missing Khmer translations?" -- a query that is natural with a
   translation table (`LEFT JOIN ... WHERE kh.id IS NULL`) but awkward with
   inline columns (`WHERE name_km IS NULL`... except that pattern doesn't
   distinguish "not yet translated" from "intentionally blank").
2. **Future locale expansion.** If XFOS expands to serve Chinese or Japanese
   tourists in Siem Reap, adding a locale is an INSERT, not an ALTER TABLE.
3. **Consistency with `menu_item_translations`.** Items already need a
   translation table (they have both `name` and `description`). Using the same
   pattern for categories keeps the codebase uniform.

**Denormalized `tenant_id` (C1):** This table carries its own `tenant_id` even
though `category_id` already points to a tenant-scoped parent. This
denormalization was a deliberate decision from the database schema stress test
(C1 finding). It allows every tenant-scoped query to filter directly on
`tenant_id` without joining to the parent, and it enables a parity trigger to
catch cross-tenant bugs at the database level.

## Part 2: CREATE TABLE

```sql
CREATE TABLE menu_category_translations (
  id          TEXT PRIMARY KEY,
  tenant_id   TEXT NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  category_id TEXT NOT NULL REFERENCES menu_categories(id) ON DELETE CASCADE,
  locale      "Locale" NOT NULL,  -- 'en' | 'kh'
  name        TEXT NOT NULL,
  UNIQUE (category_id, locale)
);

CREATE INDEX ON menu_category_translations (tenant_id);
```

**Parity trigger** (from `20260410_mvp_hardening.sql`):

```sql
CREATE OR REPLACE FUNCTION menu_category_translations_tenant_parity() RETURNS trigger AS $$
DECLARE parent_tenant TEXT;
BEGIN
  SELECT tenant_id INTO parent_tenant FROM menu_categories WHERE id = NEW.category_id;
  IF parent_tenant IS NULL THEN
    RAISE EXCEPTION 'menu_category_translations: parent category % not found', NEW.category_id;
  END IF;
  IF parent_tenant <> NEW.tenant_id THEN
    RAISE EXCEPTION 'menu_category_translations: tenant_id mismatch (row=%, parent=%)',
      NEW.tenant_id, parent_tenant;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER menu_category_translations_tenant_parity_trg
  BEFORE INSERT OR UPDATE OF tenant_id, category_id ON menu_category_translations
  FOR EACH ROW EXECUTE FUNCTION menu_category_translations_tenant_parity();
```

## Part 3: Column-by-Column

### `id` -- TEXT PRIMARY KEY

| Property | Value |
|---|---|
| Type | `TEXT` |
| Nullable | No |
| Default | Application-generated `cuid()` |
| Constraints | `PRIMARY KEY` |

Unique identifier for this translation row. Generated by the application via
Prisma's `@default(cuid())`.

**Why a surrogate key instead of `(category_id, locale)` as the PK:** While
the combination of `category_id + locale` is unique, XFOS standardizes on
TEXT cuid PKs for every table. This keeps the Prisma client API consistent
(every model has `.id`) and avoids composite-PK complexity in the ORM layer.
The uniqueness of the natural key is enforced by the `UNIQUE(category_id,
locale)` constraint separately.

---

### `tenant_id` -- TEXT NOT NULL

| Property | Value |
|---|---|
| Type | `TEXT` |
| Nullable | No |
| Default | None -- set by application |
| Constraints | `NOT NULL`, `REFERENCES tenants(id) ON DELETE CASCADE` |

Denormalized tenant identifier. MUST match the `tenant_id` of the parent
`menu_categories` row. Enforced by the `menu_category_translations_tenant_parity`
trigger at the database level.

**Why denormalize `tenant_id` here (C1):**

Without this column, the storefront query to load a menu would need:

```sql
-- WITHOUT denormalized tenant_id: requires JOIN to parent
SELECT mct.name, mct.locale
FROM menu_category_translations mct
JOIN menu_categories mc ON mc.id = mct.category_id
WHERE mc.tenant_id = 'clxyz123abc'
  AND mc.deleted_at IS NULL
  AND mc.is_active = TRUE;
```

With the denormalized column:

```sql
-- WITH denormalized tenant_id: direct filter, faster
SELECT mct.name, mct.locale
FROM menu_category_translations mct
WHERE mct.tenant_id = 'clxyz123abc'
  AND mct.category_id IN (
    SELECT id FROM menu_categories
    WHERE tenant_id = 'clxyz123abc'
      AND deleted_at IS NULL
      AND is_active = TRUE
  );
```

More importantly, it provides a defense-in-depth guarantee: even if the
application has a bug that constructs a translation INSERT with the wrong
`tenant_id`, the parity trigger will reject it before it reaches disk.

---

### `category_id` -- TEXT NOT NULL

| Property | Value |
|---|---|
| Type | `TEXT` |
| Nullable | No |
| Default | None -- set by application |
| Constraints | `NOT NULL`, `REFERENCES menu_categories(id) ON DELETE CASCADE`, part of `UNIQUE(category_id, locale)` |

The parent category this translation belongs to.

**Why NOT NULL:** A translation without a parent category is meaningless.

**Why ON DELETE CASCADE:** When a category is hard-deleted (or when a tenant
cascade reaches it), its translations must be cleaned up automatically. Note
that categories are normally soft-deleted (`deleted_at`), not hard-deleted,
so this cascade is a safety net rather than the normal deletion path.

---

### `locale` -- "Locale" NOT NULL

| Property | Value |
|---|---|
| Type | `"Locale"` (Postgres enum: `'en'`, `'kh'`) |
| Nullable | No |
| Default | None -- set by application |
| Constraints | `NOT NULL`, part of `UNIQUE(category_id, locale)` |

Locale code. MVP supports two values: `'en'` (English) and `'kh'`
(Khmer). Stored as a Postgres `"Locale"` enum and also validated at the API
boundary by a Zod enum.

**Why `UNIQUE(category_id, locale)` and not `UNIQUE(tenant_id, category_id,
locale)`:** `category_id` already implies a specific tenant (enforced by the
FK and parity trigger). Adding `tenant_id` to the unique constraint would be
redundant. The constraint says: "one translation per locale per category" --
which is the correct business rule.

---

### `name` -- TEXT NOT NULL

| Property | Value |
|---|---|
| Type | `TEXT` |
| Nullable | No |
| Default | None -- set by application |
| Constraints | `NOT NULL` |

The localized display name of the category. Examples:

| locale | name |
|---|---|
| `en` | Noodle Soups |
| `kh` | គុយទាវ |
| `en` | Grilled Meats |
| `kh` | សាច់អាំង |
| `en` | Drinks |
| `kh` | ភេសជ្ជៈ |

**Why NOT NULL:** A translation row with no name serves no purpose. If a
merchant has not yet translated a category to Khmer, the row should not exist
at all (the application falls back to the English name).

**Why no `description` column (unlike `menu_item_translations`):** Categories
are navigational labels -- "Soups", "BBQ", "Drinks". They do not need
descriptive text. Items, by contrast, benefit from a description ("Slow-grilled
pork ribs with tamarind glaze, served with sticky rice").

## Part 4: Indexes

### `menu_category_translations_tenant_idx` -- `(tenant_id)`

```sql
CREATE INDEX ON menu_category_translations (tenant_id);
```

**What queries it serves:** Any query that loads translations scoped by tenant.
The most common is the storefront menu load, which fetches all categories and
their translations for a single tenant in one query.

**Example:**

```sql
-- Load all category translations for a tenant in the customer's locale
SELECT mct.category_id, mct.name
FROM menu_category_translations mct
WHERE mct.tenant_id = 'clxyz123abc'
  AND mct.locale = 'kh';
```

### `menu_category_translations_category_id_locale_key` -- `UNIQUE(category_id, locale)`

```sql
UNIQUE (category_id, locale)
```

**What queries it serves:** This is both a constraint and an implicit unique
index. It serves:

1. **Upsert on translation save.** When a merchant saves a category name, the
   API uses `INSERT ... ON CONFLICT (category_id, locale) DO UPDATE` to create
   or update the translation atomically.
2. **Single translation lookup.** `WHERE category_id = ? AND locale = ?` is an
   index-only scan on this unique index.

**Example:**

```sql
-- Upsert a Khmer translation for a category
INSERT INTO menu_category_translations (id, tenant_id, category_id, locale, name)
VALUES ('clnew123', 'cltenant456', 'clcat789', 'kh', 'គុយទាវ')
ON CONFLICT (category_id, locale) DO UPDATE SET name = EXCLUDED.name;
```

## Part 5: Relationships

### Parent: `tenants`

```
menu_category_translations.tenant_id → tenants.id (ON DELETE CASCADE)
```

Denormalized FK for tenant isolation. Deleting the tenant cascades through.

### Parent: `menu_categories`

```
menu_category_translations.category_id → menu_categories.id (ON DELETE CASCADE)
```

The category this translation belongs to. Deleting the category (hard delete)
removes all its translations.

### Parity trigger: `menu_category_translations_tenant_parity`

Fires `BEFORE INSERT OR UPDATE OF tenant_id, category_id`. Looks up the parent
category's `tenant_id` and raises an exception if they do not match. This is
the database-level enforcement of the C1 denormalization rule -- it prevents
cross-tenant data bleed even if the application layer has a bug.

## Part 6: Real-World Usage Scenarios

### Scenario 1: Noodle stall adds bilingual categories during onboarding

"Kuy Teav Phnom Penh" registers on XFOS. The merchant portal onboarding wizard
prompts for category names in English (required) and Khmer (optional). The owner
enters:

| Category | English (en) | Khmer (kh) |
|---|---|---|
| cat-001 | Noodle Soups | គុយទាវ |
| cat-002 | Side Dishes | មុខម្ហូបផ្សេងៗ |
| cat-003 | Beverages | ភេសជ្ជៈ |

This produces 3 `menu_categories` rows and 6 `menu_category_translations`
rows. When a customer scans the QR and their phone locale is `kh`, the
storefront shows "គុយទាវ", "មុខម្ហូបផ្សេងៗ", "ភេសជ្ជៈ". When the locale is
`en`, it shows "Noodle Soups", "Side Dishes", "Beverages".

### Scenario 2: English-only bubble tea shop skips Khmer translation

"BoBa Express" in the central business district caters primarily to expats and
English-speaking locals. The owner creates categories but does not enter Khmer
translations:

| Category | English (en) | Khmer (kh) |
|---|---|---|
| cat-010 | Milk Tea | -- |
| cat-011 | Fruit Tea | -- |
| cat-012 | Toppings | -- |

This produces 3 `menu_categories` rows and only 3 `menu_category_translations`
rows (English only). The storefront's i18n layer falls back: when a
Khmer-speaking customer visits, the UI shows the English name because no `kh`
row exists.

The `setup_progress.translations_complete` flag remains `FALSE` for this tenant
until they add Khmer translations, prompting a reminder in the merchant portal
dashboard.

### Scenario 3: Merchant corrects a Khmer category name

"Angkor BBQ" realizes their Khmer category name for "Appetizers" has a typo.
The merchant portal sends an upsert:

```sql
INSERT INTO menu_category_translations (id, tenant_id, category_id, locale, name)
VALUES ('clnew999', 'cltenant-angkor', 'clcat-appetizers', 'kh', 'មុខម្ហូបញ៉ាំលេង')
ON CONFLICT (category_id, locale) DO UPDATE SET name = EXCLUDED.name;
```

The existing Khmer row is updated in place. The storefront reflects the
correction on the next page load. No category structural data is affected.

## Part 7: Design Decisions

### Denormalized `tenant_id` (C1 stress test finding)

This is the single most important design decision on this table. The C1 finding
from the database schema stress test established that every child table in a
multi-tenant system must carry its own `tenant_id` for two reasons:

1. **Query performance.** Application-layer tenant isolation means every query
   includes `WHERE tenant_id = ?`. Without a local `tenant_id`, every
   translation query must join to `menu_categories` just to filter by tenant.
   The denormalized column eliminates that join.

2. **Defense in depth.** The parity trigger (`menu_category_translations_tenant_parity`)
   catches bugs where the application constructs an INSERT or UPDATE with a
   `tenant_id` that does not match the parent category's tenant. Without this
   trigger, a coding error could silently write a Tenant A translation into
   Tenant B's category -- a multi-tenancy violation that would be extremely
   hard to detect after the fact.

**Trade-off:** Denormalization means the `tenant_id` is stored redundantly.
For a translation table with dozens of rows per tenant, the storage overhead
is negligible.

### `UNIQUE(category_id, locale)` -- one translation per locale per parent

This constraint is the core i18n invariant: a category has at most one name
per locale. If the constraint were missing, two Khmer translations could
coexist for the same category, and the storefront would not know which one to
display.

### No `created_at` / `updated_at` timestamps

Translation rows are lightweight -- they hold a name string and exist to serve
the parent category. Their lifecycle is tied to the parent's. The parent
category's `updated_at` is sufficient for cache invalidation ("has anything
about this category changed since my last fetch?"). Adding timestamps to the
translation table would increase write amplification for minimal benefit.

### No soft delete

Translations do not have a `deleted_at` column. Deleting a translation (e.g.,
removing the Khmer name for a category) is a hard DELETE. The rationale:

- Translations are easily re-created. There is no complex state to preserve.
- Soft-deleting a translation would mean the application must filter
  `WHERE deleted_at IS NULL` on every read -- extra complexity for a row type
  that is trivially reconstructable.
- If the parent category is soft-deleted, its translations remain in the
  database (soft delete does not cascade). They become visible again if the
  category is restored.
